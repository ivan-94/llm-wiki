# 问题定义实践框架

## Thesis

好的解题流程不是从答案开始，而是从定义问题开始：看见现象后，先确认差距、主人、重述方式和方案副作用，再进入方案设计。

## Framework

1. 看见现象：先描述现实，不急着归因。（[[concepts/问题三元模型|问题三元模型]]）
2. 定义差距：现实状态与期望状态之间是否有明确落差。（[[concepts/问题三元模型|问题三元模型]]）
3. 找问题主人：谁受影响、谁在意、谁有动力和能力推动改变。（[[concepts/问题主人|问题主人]]）
4. 拆掉预设方案：把“要上系统/要降价/要机器人”重述成目标或待验证假设。（[[concepts/第一问题定义陷阱|第一问题定义陷阱]]）
5. 重述问题：换主体、目标、尺度、时间和边界。（[[concepts/问题重述|问题重述]]）
6. 评估副作用：方案会改变谁的行为，把成本转移给谁，短期长期是否一致。（[[concepts/方案副作用|方案副作用]]）
7. 决定方案：在多个成立答案之间做取舍，而不是迷信第一个正确答案。（[[concepts/反过早收敛|反过早收敛]]）

## Agent 任务澄清清单

把上面的框架压缩成给 Agent 下任务前的一页检查表。它和 [[concepts/Spec-Driven Development|Spec-Driven Development]] 的“先写规格再实现”互补：问题定义负责“解对的问题”，SDD/[[concepts/Mini Spec|Mini Spec]] 负责“把对的问题写成可执行规格”。

- [ ] 现象 vs 问题：用户说的是现象、情绪、归因还是方案？（[[concepts/第一问题定义陷阱|第一问题定义陷阱]]）
- [ ] 差距：现实状态和期望状态分别是什么，落差是否明确？（[[concepts/问题三元模型|问题三元模型]]）
- [ ] 问题主人：谁最痛、最在意、最能验收这次改动？（[[concepts/问题主人|问题主人]]）
- [ ] 方案句改问题句：把“做 X”重述成“要达成什么 / 为什么 X”。（[[concepts/问题重述|问题重述]]）
- [ ] 副作用：这次改动会改谁的行为、转移什么成本、最坏后果是什么？（[[concepts/方案副作用|方案副作用]]）
- [ ] 反过早收敛：当前方案回答的是哪个问题、前提是什么、还有没有其他成立答案？（[[concepts/反过早收敛|反过早收敛]]）
- [ ] 写进规格：把澄清后的问题、约束和验收标准落进 Mini Spec / SDD，再让 Agent 执行。

## Application To Agent Work

- 给 Agent 下任务前，先把“方案句”重述成“问题句”。
- 如果用户只给出第一个问题定义，先查现象、目标、问题主人和约束。
- HAT/验收标准应覆盖方案副作用，而不只覆盖主流程成功。
- 多 Agent 并发前，先确认每个 slice 的问题主人和完成证据，避免执行错方向。
