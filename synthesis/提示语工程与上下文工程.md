# 提示语工程与上下文工程

## Thesis

提示语工程解决"如何把任务说清楚"，上下文工程解决"模型执行任务时应该看见什么"。在短任务里两者可以分开；在 Agent、长上下文和工具调用场景里，提示语只是上下文工程的一部分。LangChain 的应用案例为这个边界提供了大量实践证据（legacy · not verified）。

## Comparison

| 维度 | 提示语工程 | 上下文工程 |
| --- | --- | --- |
| 核心对象 | 指令、示例、约束、输出格式、评估标准 | 指令、记忆、工具、知识、状态、历史、工具反馈、轻量标识符 |
| 主要目标 | 提高任务接口清晰度和可迭代性 | 管理长期、多轮、带工具推理的可见信息 |
| 主要风险 | 歧义、格式不稳、边界不清 | 投毒、干扰、混淆、冲突、腐烂 |
| 常用手段 | 角色、任务、few-shot、rubric、输出 schema、prompt chaining、工具调用约束 | 写入、选择、压缩、隔离、JIT 检索、结构化笔记 |
| 验证方式 | 样例测试、prompt 对比 | 轨迹评估、上下文 ablation、线上/离线评估 |

## Langchain 案例中的实践证据（Legacy）

以下模式展示了提示语工程和上下文工程如何在实践中合流（来自 `Langchain 案例学习.xmind`，legacy · not verified）：

- **RAG 问答**：系统提示语定义助手行为（提示语工程），RAG 检索和历史改写控制模型看什么（上下文工程）。
- **History-aware RAG**：先用 LLM 改写追问（提示语技术），再控制检索上下文（上下文工程）。History-aware 改写步骤同时属于两个范畴的交叉点。
- **结构化提取**：schema 设计是提示语工程（任务接口），长文本分块策略是上下文工程（控制模型看什么）。
- **ReAct Agent**：系统提示语定义工具调用格式（提示语工程），AgentExecutor 的循环历史、工具反馈和错误摘要是上下文工程。
- **SQL 问答**：prompt 定义表结构和 SQL 格式约束（提示语工程），选哪些表、哪些高基数列值进入 prompt（上下文工程）。

## 评估闭环行

- 提示语工程的迭代需要固定数据集和评估器：改了 prompt，必须用同一批数据重新测量，才能知道是提升还是回退。
- 上下文工程的评估更难：需要控制"不同上下文策略对输出质量的影响"（上下文 ablation），而不只是比较 prompt 版本。
- Langfuse/LangSmith 提供了把两类迭代都纳入统一实验框架的工具路径（详见 [[concepts/LLMOps|LLMOps]]）。
- 评估闭环是提示语/上下文优化的"科学方法"：不能只靠感觉，必须有可重复的实验。

## Implications

- 学 prompt 时不应停在模板收集，要抽取结构：任务接口、上下文输入、输出标准和失败处理。
- 做 Agent 时不应只调系统提示语，还要控制工具数量、历史摘要、记忆检索和中间结果污染。
- prompt 优化和上下文优化都需要评估闭环，否则很难区分真实提升和偶然样例通过。
- 当 prompt 进入 Agent 场景，它会变成上下文工程的一部分：工具 schema、错误摘要、状态视图和人类协作入口都需要进入同一设计。

## My Take

当前更稳的学习顺序是：先掌握提示语结构，再学习上下文失效模式，随后用 LLM 评估把 prompt/上下文策略变成可比较实验。LangChain 案例（legacy）提供了大量"两者合流"的实践模板，即使 API 过时，设计模式仍有学习价值。

## Open Questions

- 如何为上下文工程设计可重复评估（context ablation），还需要更多具体实践案例。
- 多智能体隔离什么时候优于单 Agent 摘要与检索，需要后续专题对比。
- LangChain 案例的 API 细节（legacy · not verified），需要官方文档核验。
