## Summary

这份 source 梳理 LLM 数据获取中的搜索与网页读取链路：搜索引擎/API 负责发现候选页面，浏览器或爬虫负责抓取页面，正文抽取工具把 HTML 转为可消费文本，开源搜索产品则提供端到端参考。

## Source Digest

导图把“搜索”拆成四个环节。第一层是搜索入口：searxng 是免费元搜索，Google/Bing 是商业搜索，Tavily、Exa、Serper 是对搜索结果面向 LLM 消费的包装。source 对 Tavily 的观察是输出更适合 LLM 阅读，包含图片描述、概要和正文，但中文正文提取可能乱码。第二层是页面抓取，列出 Playwright Python 和 Crawl4AI，其中 Crawl4AI 被描述为基于 LLM 的爬虫和页面抓取引擎。第三层是正文提取，Python 侧列出 Trafilatura 和 newspaper，JavaScript 侧列出 Readability.js。第四层是开源产品参考，如 Morphic 和 MindSearch。

这份 source 对知识库的贡献是建立“发现 -> 抓取 -> 正文抽取 -> LLM 消费”的数据获取流水线，而不是把搜索工具简单列成清单。它也提醒：搜索 API 的摘要不一定足够，很多场景仍要做页面抓取和正文抽取；面向中文内容时需要单独检查编码和抽取质量。

## Key Claims

- explicit: Google 搜索每天有免费额度限制，返回概要信息较少，不方便 LLM 一次性阅读，通常还需要页面爬取。
- explicit: Tavily 输出更适合 LLM 消费，包含图片、描述、概要和正文，但中文正文提取可能出现乱码。
- explicit: 页面抓取工具候选包括 Playwright Python 和 Crawl4AI，正文提取工具候选包括 Trafilatura、newspaper 和 Readability.js。
- explicit: Morphic 和 MindSearch 被列为开源搜索产品参考。
- inferred: LLM 搜索系统应分层设计为搜索发现、页面抓取、正文抽取、结构化返回，而不是依赖单一搜索 API。
