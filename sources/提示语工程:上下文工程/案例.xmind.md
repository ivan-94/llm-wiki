## Source Digest

该 source 的核心价值不是提供一组可直接复用的完整 prompt，而是展示成熟系统提示语如何拆分行为边界、能力边界、输出习惯和风险处理规则。

Anthropic/Claude 示例体现出几个结构化设计点：先明确助手身份、当前日期和知识库边界，再描述能力范围、不可执行事项、复杂任务处理方式、争议话题处理方式、引用风险、长任务分段策略和代码输出习惯。与普通任务 prompt 相比，系统 prompt 更像一组长期适用的行为契约。

Claude 3.5 Sonnet 示例还展示了“模型不能做什么”的表达方式，例如无法打开链接时应让用户粘贴内容；遇到非常冷门信息或引用文献时应提示可能幻觉；不能执行任务时直接说明，而不是用道歉填充。这些规则可编译成提示语工程中的“边界声明”和“失败处理”模式。

图像相关说明体现了安全和隐私约束如何写入系统提示：面对包含人脸的图片时，不识别或命名具体个人；如果用户提供身份信息，可以讨论该名字对应的人，但不确认图中身份。它也要求先复述图中指令再继续，说明多模态 prompt 需要处理“图像内指令”和用户指令的关系。

“汉语新解”案例使用类 Lisp/代码结构组织角色、风格、函数、few-shot 和输出 SVG 卡片。这说明 prompt 可以借用程序结构来表达稳定流程，尤其适合固定输入、固定变换、固定输出样式的创作任务。

这张图对 wiki 的贡献在于提供可归纳的 prompt 案例模式：身份与边界、能力与不可做事项、输出格式、失败处理、多模态安全、风格化函数式提示语。

## Key Claims

- 成熟系统提示语通常包含身份、时间、知识边界、能力范围、拒绝/失败处理、输出习惯和安全规则。
- “不能做什么”需要明确写入提示语，否则用户预期与模型能力容易错位。
- 多模态提示语需要显式处理图像安全、隐私和图像内指令。
- 类代码结构可以帮助把创作型 prompt 固化为角色、函数、样例和输出模板。
- 案例学习的重点是抽取结构模式，而不是机械复制完整提示语。
