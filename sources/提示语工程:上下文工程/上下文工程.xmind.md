## Summary

这个 workbook 用两个 sheet 组织上下文工程：`上下文工程` sheet 解释定义、失效模式和四类策略；`Anthroipic 实践` sheet 补充面向 Agent 的提示语复杂度、JIT 检索、压缩、结构化笔记和子代理/多代理实践。

## Source Digest

该 source 的主线是“为什么需要上下文工程 -> 它和提示词工程有什么区别 -> 上下文由什么组成 -> 可以用哪些策略管理上下文 -> Anthropic/Claude Code 类实践如何落地”。

它首先从长上下文本身的失效模式解释需求：上下文窗口变长并不等于性能稳定，成本和延迟会上升，模型对信息的回忆、取舍与冲突处理也会变差。图中把常见问题分成上下文投毒、干扰、混淆、冲突和腐烂。它们共同指向一个实践判断：只要信息进入上下文，模型就不得不分配注意力；无关、冗余、错误或互相矛盾的信息都会污染任务执行。

第二层论证来自智能体复杂度。智能体不仅消费用户输入，还会携带系统指令、工具定义、MCP、外部数据、历史消息、工具反馈和中间尝试。生产可用的智能体要长时间可靠运行，就必须控制错误和噪声在轨迹中累计。

source 将提示词工程与上下文工程区分开：提示词工程偏向编写和组织指令，优化单次或短期交互；上下文工程偏向策划整个推理过程中进入有限窗口的所有信息。提示词仍是上下文的一部分，但上下文还包括记忆、工具、知识、状态和历史。

source 使用“LLM 是 CPU、上下文窗口是 RAM”的比喻来帮助理解。上下文工程就是在智能体轨迹的每一步，用恰当的信息填满工作记忆，而不是把所有可得信息都塞进去。

策略部分分为四类：

- 写入上下文之外：把草稿、状态和长期记忆放在窗口外，需要时再检索。草稿偏单次会话，长期记忆跨会话，可由反思或总结产生。
- 选择上下文：从草稿、长期记忆、工具库和知识库中挑选当前任务所需内容。图中特别提到工具过多会降低性能，可以用 RAG 或其他检索机制做工具选择。
- 压缩上下文：通过摘要、递归摘要、分层摘要和裁剪减少 token，只保留完成任务必要的信息。Claude Code 的 Auto Compact 被用作例子。
- 隔离上下文：通过多智能体、环境/沙箱和结构化状态隔离不同任务或子问题，让每个上下文保持边界清晰。

第二个 sheet 把这些原则落到更具体的 Agent 实践上：提示语应该在过度复杂和过度模糊之间取得平衡；少量高质量、多样化示例优于穷举边缘情况；prompt 开发应从最小提示开始，反复测试失败模式，再有针对性补指令和示例。信息检索方面，它强调 JIT（即时）检索：上下文中保留轻量标识符，如文件路径、数据库查询或网页链接，需要时再通过工具动态加载内容。长周期任务方面，它把压缩、结构化笔记和子代理/多代理列为控制上下文膨胀和分工隔离的手段。

这张图对当前 wiki 的价值在于，它给出了一个可复用分类框架：上下文问题不是“窗口够不够大”，而是“什么该进、什么该留在外部、什么该摘要、什么该隔离，以及 Agent 如何在运行中维护这些边界”。

## Key Claims

- 长上下文会带来成本、延迟和可靠性问题，不能把“扩大窗口”视为上下文管理的完整解法。
- 上下文投毒、干扰、混淆、冲突和腐烂分别描述了错误信息、过量信息、冗余信息、矛盾信息和注意力退化造成的失效模式。
- 生产级智能体需要管理整个上下文状态，而不是只优化一段 prompt。
- 上下文工程可以按写入、选择、压缩、隔离四类策略来组织。
- 工具定义、工具反馈、记忆、知识和状态都属于上下文管理对象。
- JIT 检索通过保留轻量标识符、运行时动态加载内容，降低把完整资料常驻上下文的成本。
- 长周期 Agent 需要压缩、结构化笔记和子代理/多代理等机制来控制历史膨胀和任务边界。
- prompt 指令复杂度需要平衡：过度硬编码会脆弱，过度模糊会不稳定。
