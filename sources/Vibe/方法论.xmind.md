## Summary

这个 XMind 把 Vibe Coding 下的软件工程方法论整理成三条主线：意图驱动的软件工程范式、Spec-Driven Development 与 context engineering 的工具化流程、以及面向 LLM 协作的 SOP/架构/测试/领域建模实践。核心判断是：AI 让代码生成变快后，真正需要强化的是规格、边界、反馈、验收和可复现的工程资产。

## Source Digest

资料先把软件工程的演进描述为抽象层不断提升：从机器码、语言结构、框架，进一步走向用自然语言、测试、草图和规格表达意图。开发者角色从直接构建实现转向制定约束、边界、契约和判例；价值从记忆语法和写样板代码，转移到系统性思考、批判性审查和领域心智模型。由于 AI 生成的代码可能短暂且不稳定，行为、测试和规格成为锚定系统正确性的核心资产。

规格驱动开发部分把 Kiro、OpenSpec、Spec Kit、SpecStory 等工具和观点纳入同一框架：先沟通和结构化意图，再形成 requirements/design/tasks 或 spec/plan/tasks 等可追溯工件，最后由 AI 生成实现并通过测试、验收和生产反馈更新规格。Spec Kit 分支特别强调规格是通用语言，代码只是表达；流程包含从模糊想法到 PRD、研究代理补充上下文、计划与代码生成、运行反馈回写规格。其“九条法则”把库优先、CLI 可观测、测试优先、简洁反抽象、集成优先测试和模板关卡作为约束 LLM 的机制。

SOP 和 AI 友好工程部分给出可执行骨架：Context Loop 明确问题、范围、约束、依赖和 golden cases；Plan Loop 切出小粒度任务、风险和回退；Scaffold Loop 先定接口/DTO/错误码与测试骨架；Implement/Test/Review/Doc Loop 通过小步提交、失败用例、回归验证、人机评审和知识回灌形成闭环。面向 LLM 的代码库应契约优先、可测试、可观测、架构简洁、技术栈主流、循环粒度小，并通过 DDD/TDD/BDD 把领域模型和行为示例变成 AI 协作的输入与验收界面。

## Key Claims

- explicit: Vibe Coding 将开发者的主要职责从写实现转向表达意图、制定约束、边界、接口契约和测试判例。
- explicit: 在 AI 参与生成代码后，代码不再是唯一真理之源；测试、规格和行为成为锚定系统正确性的关键资产。
- explicit: 规格驱动开发要求从模糊想法沉淀 PRD/spec，再生成计划、任务、代码、数据模型和测试用例。
- explicit: Spec Kit 的开发原则包括库优先、CLI 接口、测试优先、简洁反抽象、集成优先测试和模板关卡。
- explicit: SOP 对 Vibe Coding 的价值在于用 gate 和 artifact 固化上下文、计划、测试、评审和文档，减少漂移与返工。
- explicit: AI 友好的工程环境强调契约优先、标准协议、快速质量反馈、结构化日志、强类型、简洁架构和可预测惯例。
- explicit: DDD、TDD、BDD 在 AI 协作中从传统工程实践升级为表达领域意图和验证行为的核心交互界面。
- inferred: 这份方法论把“Vibe”从随意提示模型生成代码，重新定义为用规格、SOP 和反馈系统治理不确定性的工程方法。
