## Source Digest

图中以“模块五：落地与反模式”为主题，强调 SbE 的价值不在于把测试写得更漂亮，而在于让需求、示例、规则与自动化形成闭环。上半部分列出四类常见失败：示例太技术化，导致业务无法参与；把示例当测试规格，忽略业务规则表达；过早自动化，制造大量脆弱场景；团队只让 QA 写示例，造成示例沦为测试文档。

中部用“反模式 vs 正确做法”对比出落地顺序：不要先写自动化、不要让 QA 单独产出、不要以“越多越好”为目标；应先讨论业务场景，再由业务、开发、测试共同创建可读、可验证、高价值示例。下半部分给出成功关键：示例必须来自真实业务、需要随规则变化持续维护、Cucumber/SpecFlow/Behave 只是辅助工具。登录功能案例显示，坏需求只说“友好登录体验”，无法验收；SbE 写法拆成账号密码正确、密码错误、连续失败锁定等可验证示例，使需求清晰、规则明确、可直接转成验收测试。

## Key Claims

- explicit: SbE 不是把测试写得更漂亮，而是让需求、示例、规则与自动化形成闭环。
- explicit: 示例如果只表达 API、字段名、状态码等技术细节，业务人员难以参与讨论，也无法确认示例是否真正代表业务规则。
- explicit: 自动化应服务于已经达成共享理解的高价值示例，过早把大量 UI 场景自动化会带来维护成本和脆弱测试。
- explicit: 成功落地 SbE 需要业务、开发、测试共同创建和维护示例，而不是由 QA 单独输出测试文档。
