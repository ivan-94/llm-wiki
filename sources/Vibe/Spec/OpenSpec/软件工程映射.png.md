## Source Digest

图像顶部明确说 OpenSpec 不是照搬某一种方法论，而是把多种成熟实践压缩成 AI 可执行的规格工作流。左半部分把 Specification by Example / Acceptance Criteria 映射到 Requirement + Scenario，强调用具体场景消除歧义但不等同于可执行测试；把 BDD / Given-When-Then 映射到 WHEN / THEN 场景结构，保留 Gherkin 思维但不强制完整 Gherkin；把 RFC 2119 映射到 SHALL / MUST 规范语气；把 lightweight RFC / change proposal 映射到 `proposal.md`；把 Design Doc / ADR 映射到 `design.md`。

右半部分把 Delivery Plan / Checklist 映射到 `tasks.md`，用于把实现拆成 AI 可执行任务；把 Git Patch / changeset / migration 映射到 delta specs，用 ADDED/MODIFIED/REMOVED/RENAMED 表达规范差异；把 Living Documentation / Contract Repository 映射到 `openspec/specs`，作为当前行为的唯一事实来源；把 Audit Trail / Decision History 映射到 `archive`，保留为什么、怎么做、做完什么；把 DDD bounded context flavor 映射到 capability-based specs，用能力边界组织规格但不是完整 DDD 框架。底部 lifecycle 是 propose -> spec/design -> tasks -> apply -> archive -> living specs。
