## Source Digest

图像用四个彩色模块表达 OpenSpec 的产物边界。蓝色的 `proposal.md` 聚焦 Why / What / Capabilities / Impact，强调 1-2 页、只写变更动机和范围，不写实现细节，并由 capabilities 决定 spec 文件。绿色的 `specs/<capability>/spec.md` 被定义为行为契约而不是实现计划，使用 ADDED / MODIFIED / REMOVED requirements，并要求 requirement 使用 SHALL 或 MUST、至少包含一个 scenario、scenario 使用 `#### Scenario`，MODIFIED 要提供完整 requirement block。紫色的 `design.md` 负责 Context、Goals / Non-Goals、Decisions、Risks / Trade-offs、Migration Plan，主要记录关键决策、风险和迁移路线，不直接写运行实现。橙色的 `tasks.md` 是可追踪实现清单，用 `- [ ]` / `- [x]`、X.Y 编号、依赖顺序和完成勾选来约束执行粒度。

这张图的学习价值在于把 OpenSpec 产物拆成“为什么要变、系统行为怎么变、技术上怎么做、具体执行什么”四层，减少把设计、实现计划和验收行为混在一个文档里的风险。它也暗示 OpenSpec 的 source of truth 不是临时变更目录，而是 apply/archive 后沉淀的规范账本。
