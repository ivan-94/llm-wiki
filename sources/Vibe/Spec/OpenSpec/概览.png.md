## Summary

这张图概览 OpenSpec 从需求到可归档规范账本的工作流：先初始化或更新工具目录，再创建 change，产出 proposal/spec/design/tasks，按任务实现并勾选，最后同步归档为 `openspec/specs/` 中的 source of truth。图中还列出 OpenSpec 的原则、项目状态文件、change artifacts、工具集成方式和校验标准。

## Source Digest

图像上半部分把 OpenSpec 工作流画成 6 步：`init / update` 初始化 `openspec/` 并生成 AI 工具指令；`propose / new` 创建 `openspec/changes/<change>/`；`artifacts` 产出 proposal、specs、design、tasks；`apply` 按 `tasks.md` 实现并勾选任务；`sync / archive` 合并 delta specs 并归档 change；最终形成 `openspec/specs/` 下的新 source of truth。箭头回路说明下一次变更基于当前 specs。

中部原则区强调 OpenSpec 是流动而非僵硬、迭代而非瀑布、轻量而非复杂、brownfield-first，并把 spec 视为可观察行为契约而不是实现细节。下半部分列出项目状态和产出标准：项目侧有 `openspec/specs/<capability>/spec.md`、`openspec/config.yaml`、`openspec/schemas/<schema>/`；change 侧有 `proposal.md`、`specs/<capability>/spec.md`、`design.md`、`tasks.md` 和 `archive/YYYY-MM-DD-<change>/`。工具集成包括 `/opsx:*` slash commands 和面向 Claude / Codex / Cursor / Gemini / Windsurf 的 skills。校验标准要求 SHALL/MUST、每个 requirement 至少一个 scenario、scenario 使用 `#### Scenario`、MODIFIED 完整 block，并在 archive 前校验 delta specs。
