## Summary

这份年中总结把 Vibe Coding 后的软件开发工作变化概括为：开发本身被极大加速，人的主要时间从编码转向共识对齐、验收、治理和 Harness 工程。它区分了简单任务、复杂任务、业务项目和严肃项目的不同工作流，并提出项目、基础设施和团队组织都需要做 Agent Friendly 改造。

## Source Digest

资料先用时间分布对比说明工作重心变化：过去主要在对齐、开发、测试、联调中以开发为主；现在更多时间放到对齐和计划、验收、治理、Harness 调优，编码本身变成监督和引导 Agent 开发。它把这种变化解释为软件工程核心命题“共识对齐”的重新分配：PRD、原型、UI、DDD、接口文档、Code Review、TDD、BDD 本来都是对齐共识的工具；当 Agent 加入开发后，需要更多 Spec、PRD、Planning、脑暴、盘问、复述等沟通工具，确保 Agent 与人对需求理解一致。

资料对治理的定义分为静态、动态和上下文三类。静态治理是定期扫描和重构 AI 高速产出的低质量代码；动态治理是通过异常监控、Sentry、GitHub issue、修复 PR、审核上线等链路支持严肃生产系统；上下文治理则属于 Harness 工程，目标是避免 AGENTS.md、Skills 和协作上下文膨胀导致生成效果下降，让 Agent 以更精准的上下文工作。Harness 工程被定义为有利于 LLM 更好、更稳定实现人类意图的一组工程设施，覆盖共识对齐、环境感知、工具扩展、反馈连接、任务执行、结果验收和项目工程化措施。

后半部分给出实践判断：要多用多折腾，Codex 是主力开发工具，Kimi/MiniMax/GLM 等可轮流用于脏活、HAT 测试、脑暴和计划；思考不能完全交给 AI；vibe 和严肃编程不是一回事，但大部分软件需求可以被 agent 直接做。作者的工作流是：简单任务直接对话 oneshot，稍复杂任务先 planning，业务项目可以 Vibe Coding，严肃项目走更完整流程。作品清单包括文字游戏外挂平台、基于 GitHub Issue 的 Agent 看板、prompt 评估器、自有 Skills 工作流、FinderX、XMind 管理器和搜索器、破解 XMind、xmind-cli。

- explicit: 结果验收不能只依赖 Agent 工具和模型升级，项目侧也需要做 Agent Friendly 改造。

- inferred: 这份总结把 Vibe Coding 的核心能力边界放在“快速生成与验证”，而把严肃生产交付的核心放在治理、验收和工程化 Harness。
