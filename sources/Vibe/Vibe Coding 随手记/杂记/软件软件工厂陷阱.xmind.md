## Summary

这份 XMind 摘录并消化《The Software Factory Trap》的核心观点：AI agent 已经能快速生产代码，但团队真正稀缺的是可验证、可追溯、可维护的可信结果。它把失败模式从“代码写不出来”转向“上下文、证据、评审、权限、组织结构和工厂本身需要被维护”。

## Source Digest

source 的中心判断是：把 AI agent 接入代码库、任务系统和测试流程，并不会自动形成高效软件工厂。代码只是系统的表面，背后还有需求来源、推导过程、产品目的、边界澄清、权衡取舍和设计历史；如果 agent 只看到代码，却被要求延续整个系统，评审者就要重新考古这些上下文。

它进一步把瓶颈命名为“可验证性”：团队不只需要知道 agent 改了哪些文件，而要知道改动来自哪个需求、依赖哪些文档、假设了什么、由哪些检查证明、哪些 source 变化会让结果过期。source 用 ontology 与 epistemology 组织这个问题：前者建模 requirement、source、task、artifact、permission、test、proof claim、decision 等对象及其关系；后者要求 proof type 匹配 claim type，例如 API 200、后端测试、截图或简短总结都只能证明有限类型的结论。

这份材料也把 agent-heavy engineering 的组织风险讲清楚：工程师可能从个人贡献者变成领域负责人，指挥一组 agent worker；局部执行速度提升后，冲突假设、重复修改、不同 proof standard 和上下文过载会回流成人类协调成本。因此它主张小而权威的上下文包、可过期性追踪、同步/异步任务区分，以及对 prompts、skills、context pipelines、review rituals、权限和 trace 的持续维护。

## Key Claims

- explicit: AI 编程工具擅长写 diff、搭服务、补测试和迁移组件，但“生成正确代码”仍依赖需求、架构、业务约束和可维护性的证明。
- explicit: 软件工厂不是 agent 本身，而是围绕 agent 的 onboarding、目标、权限、责任边界、任务定义、review loop、escalation path 与 definition of done。
- explicit: proof type 必须匹配 claim type；错误类型的证据会让系统看起来严谨，但实际上证明了错误的东西。
- explicit: 更多上下文不等于更好结果；如果 agent 的输出可能依赖一切，staleness detection 几乎不可行。
- inferred: 这份 source 可作为后续编译“Agent 软件工厂”“上下文工程”“HAT/验收证明”相关概念的核心论据。
