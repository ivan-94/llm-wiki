## Summary

这张图把 AI Coding 初期的软件创建拆成“先产品化，后工程化”：先证明事情值得做，再把它做严肃。它区分 Vibe Coding 和严肃 Coding 的阶段目标，并把产品经理、工程师与长期 Builder 能力放在一条演进线上。

## Source Digest

图的主张是，代码变便宜以后，稀缺资源从“写代码”转向“判断”。第一层判断是产品判断：有没有人要、是否值得做、是否能先验证。第二层判断才是工程判断：架构、数据模型、安全、测试、可靠性、长期维护和真实业务承载。

图中把 Vibe Coding 描述为探索、验证、找感觉的模式，关注问题、体验和核心流程，允许粗糙代码、临时逻辑和快速重写，标准是有人愿意用、愿意反馈、方向成立。严肃 Coding 则面向交付、上线和长期维护，关注架构、数据模型、安全、测试、可靠性和可扩展，标准是稳定可用、可演进、可承载真实业务。图中特别强调二者不是高低之分，而是阶段不同。

流程对比也服务于这个结论：传统思路是先想清需求、设计架构、写代码、打磨质量、发布；AI Coding 时代更偏向快速具象化、获取反馈、验证问题，再工程化重做。现实分工上，产品经理更适合早期 Vibe Coding，负责找场景、定义假设、快速原型、验证需求和判断值不值得做；工程师更适合严肃 Coding，负责识别长期风险、重构/重写、设计系统边界、补齐测试安全可靠性，并把 demo 变成产品。长期趋势是产品和工程能力互补，最强的 Builder 能在两种思维之间切换。
