## Summary

这张图讨论 Vibe Coding 中“具体化 vs 泛化”的约束位置：问题不在于取中间值，而在于目标必须像钉子一样具体，路径可以像水一样流动。它反对把 Agent 限制成脚本解释器，也反对给过度自由导致目标漂移；主张具体化 outcome，泛化 process。

## Source Digest

图中先指出写 Skills、Planning、Specs 时总会遇到具体化与泛化的 trade-off。过度具体化会限制模型创造力和通用解法，让 Agent 变成脚本解释器，抑制探索与创意，并导致场景变化时复用失败；过度泛化则给了自由却丢失方向，造成目标漂移、验收不稳定、上下文膨胀和探索过多。

核心观点是不要追求折中，而要把约束放在正确层级：Outcome 需要具体化，Process 需要泛化。应该具体化任务意图、边界条件、验收标准、风险约束和输出格式；应该泛化实现路径、局部技术选择、探索顺序和修复策略。

图中给出四层可执行框架：L1 意图层回答为什么做，L2 边界层说明不能偏到哪里，L3 策略层推荐怎么做，L4 操作层只在必要时规定步骤。越靠上越应该具体，越靠下越应该留白。落地原则是担心跑偏就具体化边界与验收，担心僵化就泛化实现路径，不要替 Agent 写剧本，但要给 Agent 设定物理法则。

## Key Claims

- explicit: Vibe Coding 的具体化与泛化不是取中间值，而是把约束放在正确的位置。
- explicit: 过度具体化的问题包括把 Agent 变成脚本解释器、抑制探索与创意、复用性差和换场景失效，结果是稳定但僵硬。
- explicit: 过度泛化的问题包括失去方向、目标漂移、验收不稳定、上下文膨胀和探索过多，结果是灵活但易跑偏。
- explicit: 图中主张 Outcome 具体化、Process 泛化；目标不能跑偏，路径可以开放。
- explicit: 四层框架为 L1 意图层、L2 边界层、L3 策略层、L4 操作层，并提示越靠上越具体、越靠下越留白。
- inferred: 该图适合沉淀为 Agent 指令编写原则：明确不可偏离的结果与约束，给实现过程保留搜索和调整空间。
