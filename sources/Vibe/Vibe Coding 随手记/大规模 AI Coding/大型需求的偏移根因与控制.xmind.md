## Summary

这份多 sheet XMind 是一套大型 AI Coding 需求偏移治理框架。它把失败根因归纳为稀疏 spec、未定义空间、决策权泄漏、决策粒度失控、证据过拟合、短视优化和工程价值函数缺失，并给出决策分层、六张治理图、完整工作流和工程宪法作为控制方法。

## Source Digest

source 的总命题是：大型需求失败通常不是 agent “不会写代码”，也不是单纯 spec 不够详细，而是自然语言 spec 只能提供稀疏约束，软件系统却处在高维行为空间中；agent 会在未定义空间自动补全，并默认优化当前可见目标，最后交付一个表面能跑但语义、集成和长期工程质量不可靠的系统。

根因分析将“决策权泄漏”放在核心位置：权限例外、状态含义、错误码、校验层级、批量入口是否复用规则等，本质上可能是产品、安全、架构或 contract 决策，却会因 spec 未显式说明而被 agent 静默写进代码。source 同时指出“暴露所有决策”也不可行，因为决策具有分形特征，所以需要管理决策分辨率：L0 目标层由人类决定，L1 语义层与 L2 边界层强治理，L3 设计层按风险过滤，L4 实现层默认交给 agent。

治理工具被组织成六张图：决策图防止关键判断泄漏，语义图固定实体、状态、规则和不变量，执行图确保规则落到所有入口的必经路径，证据图为关键语义匹配测试和 contract，反捷径图审查哪些错误实现也能通过当前测试，维护者图检查实现未来改起来是否痛。完整工作流从原始需求开始，依次设定决策分辨率、生成各类治理图、读取工程宪法、测试先行、实现、反捷径审查、维护者审查、必要重构，最后用功能、语义、决策归属、执行路径、证据强度、抗捷径能力和可维护性共同定义完成。

工程宪法 sheet 将长期工程质量从 feature spec 中抽离为项目级稳定约束：优先复用现有抽象、业务规则放在正确层、单一事实来源、语义命名、最小可维护实现、开放已知变化点、避免幻想式抽象；同时禁止为单个测试写特判、复制业务规则、绕过 service 直接写库、创建平行类型、用 broad catch 掩盖错误、让前端成为安全唯一边界、轻易引入依赖和大范围无关重构。

## Key Claims

- explicit: 大型需求的失败来自稀疏 spec、未定义空间、决策权泄漏、决策粒度失控、证据过拟合、短视优化和工程价值函数缺失的叠加。
- explicit: 决策权泄漏是指本该由人类、产品、架构师或领域规则决定的事项，被 agent 在实现过程中静默决定。
- explicit: 决策分辨率需要按 L0-L4 分层治理；L1 语义层和 L2 边界层必须强治理，普通 L4 实现细节默认放权。
- explicit: “测试通过”只是代理信号，可能被 agent 过拟合，不能等同于语义完成。
- explicit: 强证据应覆盖规则族、组合空间、破坏性验证、参照系对比和对抗性审查。
- explicit: 大型 coding agent 的完成定义应包含行为实现、语义证明、决策受控、粒度正确、路径收敛、测试抗投机、代码可维护。
- inferred: 这套框架把 AI Coding 从“写代码任务”提升为“决策治理 + 语义建模 + 证据工程 + 工程价值函数”的协同过程。
