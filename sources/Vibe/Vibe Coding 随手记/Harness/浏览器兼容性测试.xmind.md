## Summary

这份 XMind 用 Selenium 构建浏览器兼容性测试的完整心智模型：Selenium 是基于 W3C WebDriver 的真实浏览器控制层，适合让 Agent、CI 和人类调试共享同一套跨浏览器验证能力。内容覆盖核心架构、部署方式、被测应用接入、版本矩阵、机器接口、人类观察接口、Agent 封装、最小工作流和长期使用边界。

## Source Digest

资料把 Selenium 兼容性测试定位为“统一接口控制真实浏览器”，而不是模拟浏览器或替代单元/组件/API 测试。基本链路是 Test Runner/Agent 调用 Selenium Client，经 WebDriver Protocol 到 Grid/Node，再驱动 Chrome、Firefox、Edge 等 browser runtime 访问 Web App。Grid 的 Router、New Session Queue、Distributor、Session Map 和 Node 共同解决多浏览器、多版本和并发调度。

部署层面分为宿主机直接运行、Standalone Browser Container、多 Standalone 容器、分布式 Selenium Grid 和远程 Grid/云测试平台。source 的实用边界很明确：本地 Selenium 适合主流桌面浏览器和轻量矩阵，Safari/iOS/Android WebView/老 IE/真机移动端更适合云真机或专用平台；矩阵应该按用户占比、风险和差异动态调整，而不是无限扩张。

Agent/CI 接口的核心是 WebDriver Endpoint、capability 和产物目录。capability 描述 browserName、browserVersion、platformName、viewport、test name、session metadata；自动化脚本执行打开 URL、点击输入、等待、执行 JS、读取 DOM/Cookie/source、截图和收集日志；产物包括截图、console logs、video、page source、JSON report 和 markdown summary。人类接口则通过 Grid UI、noVNC/VNC 观察和接管运行中的 session，形成 agent 准备状态、人类检查复现、agent 抓日志并汇总报告的混合工作流。

面向 agent 的封装建议是 `compat-runner`：通过一条命令屏蔽 Grid 细节，用 `matrix.yml` 管理浏览器矩阵、viewport、URL、用例和产物路径，并为每次运行单独保存 artifacts。最小流程是启动 Selenium/Grid/Node，启动或指定被测应用，读取 matrix，为每个浏览器创建 session，执行统一脚本，保存证据，标记 pass/fail/uncertain，汇总跨浏览器差异并输出人类可读报告。

## Key Claims

- explicit: Selenium 是跨浏览器自动化控制层，基于 W3C WebDriver 标准协议，控制真实浏览器而不是模拟浏览器。
- explicit: Selenium 本身不提供浏览器，需要 browser 与 driver 配合，可本地运行也可接入 CI/Agent 工作流。
- explicit: Selenium Grid 通过 Router、New Session Queue、Distributor、Session Map 和 Node 调度多浏览器、多版本和并发执行。
- explicit: 被测应用接入应保证 URL 稳定、测试数据可重复、环境差异显式记录，不能把本机偶然可访问当作可复现能力。
- explicit: 现代主流版本优先使用官方 Selenium/browser 镜像并固定 tag，同时记录 browser、driver 和 selenium version。
- explicit: 本地 Selenium 适合主流桌面浏览器和自动化兼容验证，不适合全平台穷举、iOS Safari、老旧 IE 或真机移动端覆盖。
- explicit: 自动化结果应保存截图、日志、页面源码、视频、JSON report 或 markdown summary，而不只输出 pass/fail。
- inferred: `compat-runner + matrix.yml + artifacts directory` 是把 Selenium 变成 agent-friendly 兼容性测试能力的最小工程抽象。
- inferred: 人类观察入口不是附加项，而是视觉和交互问题调试中的关键验收接口。
