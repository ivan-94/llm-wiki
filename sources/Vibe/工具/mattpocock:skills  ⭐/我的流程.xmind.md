## Summary

这份 XMind 是一套个人 Agent 工程流程：从项目初始化、需求共识、PRD/issue 拆解、任务分诊、TDD/诊断/交叉 review/PR/HAT，到治理、并发执行模式、GitHub 依赖和长期组织提效洞察。

## Source Digest

流程从初始化开始：`/setup-matt-pocock-skills` 配置基础 skills 与上下文，`/setup-agent-workflow` 为项目定制工作流和交接规范，`/setup-agent-runtime` 构建人机共驾开发环境。进入需求阶段时，source 强调用 `/grill-me` 或 `/grill-with-docs` 前置挖掘需求分支，并把重复对齐的问题沉淀到文档、ADR、CONTEXT 或 CLAUDE/AGENTS 引用中；需求仍不清楚时，用 `/prototype` 做一次性原型来降低决策风险。

计划和执行阶段先用 `/to-prd` 形成详细计划，再用 `/to-issue` 拆 vertical slice，并让 AI 分析串行/并发依赖和适合 LLM、人类审查的粒度。任务进入 GitHub 后，通过 `/triage` 决定哪些 issue 可交给 Agent，哪些需要人工；实现侧可以用 `/tdd`、`/diagnose`、`/cross-review`、`/create-pr`，也可以用 `/deliver-issue` 或 `/deliver-prd` 做一条龙交付。验收侧包括 `/hat-prepare`、`/hat-run`、`/hat-dispatch`、人工验收和 `/hat-copilot`，最后进入 `/merge-pr`。

治理与洞察部分把 skills 看作新的复用单位：以前复用代码，现在复用共识与流程；Agent 抢走编码后，人类主要成本转向需求对齐和流程治理。source 还把 GitHub 解释为 issue/PR/评论/标签/CI/review 的成熟协作平台，适合作为 Agent 队列和云端一体入口。并发执行模式中，父 Agent 负责探索、规划、编排 sub-agent、验收合并和 cross-review；子 Agent 在隔离 worktree 中负责具体 slice 的 TDD。

- inferred: 该流程把“需求共识 -> 可执行 issue -> 可验收交付 -> 流程反思”作为个人 Agent 协作的主循环。
