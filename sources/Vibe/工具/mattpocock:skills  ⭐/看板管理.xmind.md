## Summary

这份 XMind 设计了一个本地 GitHub 项目的 Agent 工作流控制台：用看板组织 Issue、PR、Label、Action、Runner 和 Terminal，把 agent 执行、HAT、分诊和人工协作流程变成可配置的桌面/CLI 工具。

## Source Digest

该工具的核心对象是 workspace、board、lane、card、action、runner 和 terminal。workspace 支持多个本地 Git 项目，自动识别 GitHub remote，并为每个工作区保存独立 workflow 配置。board 分为 Issue 看板和 PR 看板：Issue 侧有 Inbox、Needs Info、Ready For Agent、Ready For Human；PR 侧有 Initial、HAT Ready、HAT Needs Human、HAT Blocked、HAT Passed。泳道不直接等同于 Label，而是由 labelsAll、labelsAny、labelsNone、noAssignee、draft、reviewDecision 等查询条件组合出来。

Action 层把泳道选择和 Agent 命令模板连接起来。默认示例包括 Inbox 的 `/triage`，Ready For Agent 的 `/deliver-issue` 和 `/tdd`，HAT Ready 的 `/hat-dispatch`；支持聚合执行或分开执行，并暴露 `{{refs}}`、`{{ref}}`、`{{count}}`、`{{itemsJson}}` 等模板变量。Runner 层内置 Codex 和 Claude Code，也允许自定义命令、显示名、参数和权限模式，执行前可以选择 runner、权限模式并预览 CLI。Terminal 层支持 Codex/Claude Code 交互式 TUI，并可选择 System Terminal、iTerm、Warp、Ghostty 的窗口或 tab 打开方式。

卡片和状态管理强调可操作性：卡片展示编号、标题、作者、assignee、branch、label、GitHub 链接，支持多选、运行状态跟踪、label 增删、局部刷新。刷新策略包含手动刷新、默认 5 分钟自动后台刷新和缓存增量更新。消息系统用 toast、消息收件箱和警告 icon 统一反馈。配置中心集中管理 workflow、runner、terminal 和 workspace。启动方式覆盖 `agent-board` CLI 和 macOS App，并在启动时读取当前目录 Git 状态、发现 GitHub remote、添加或切换 workspace。

## Key Claims

- explicit: 看板工具定位为本地 GitHub 项目的 Agent 工作流控制台，而不是通用项目管理工具。
- explicit: Issue/PR 泳道与 GitHub Label 解耦，泳道可以由多种查询条件组合，而不是要求用户预先创建所有推荐 label。
- explicit: Action 可以按泳道配置，并通过模板变量把选中 Issue/PR 注入到 `/triage`、`/deliver-issue`、`/tdd`、`/hat-dispatch` 等命令。
- explicit: Runner 抽象需要同时支持 Codex、Claude Code、自定义 CLI、权限模式选择和执行前 CLI 预览。
- explicit: Terminal 需要保留交互式 TUI，让用户在启动 agent 后继续与 Codex CLI 或 Claude Code CLI 交互。
- inferred: 这个看板是把 triage、delivery、HAT、review 等 agent workflow 产品化为“可视状态机 + 命令启动器”的设计草案。
