## Source Digest

source 的核心观点是：Agent 记忆不是“把聊天记录塞进向量库”，而是一套跨存储、抽取、检索、压缩、演化、治理和人机协作的长期系统。记忆存储可以是向量库、文件系统、图数据库或混合方案；处理层负责嵌入、摘要和分段；检索层需要多阶段召回、重排与上下文注入；控制层要处理生命周期、冲突、资源预算和 PII。按用途看，记忆可分为 semantic facts、episodic traces、procedural rules；按时间看，短期记忆更像 thread/session 的运行状态，长期记忆则需要持久化、检索、压缩、筛选和更新。

source 把记忆系统的难点归纳为四组：压缩不是普通摘要，而是决定抽象层级、触发时机和信息保真；演化要求 append、replace、delete 和多版本时间语义；冲突解决需要时间、置信度、来源和用户直接陈述优先级；衰减要减少检索污染和记忆爆炸。各方案体现了不同取向：QMD 偏本地 RAG；Letta 使用常驻上下文的白盒记忆块；mem0 同时处理普通向量记忆、程序性记忆和图记忆；OpenViking 和 memU 采用类文件系统/层级目录范式；LanceDB Pro 强调六类抽取、混合检索和 Weibull 衰减；Graphiti 通过时序图和溯源管理事实；Hindsight 强调 retain/recall/reflect 的学习闭环；Claude Code、ChatGPT、Hermes 展示了产品级记忆在文件、会话、摘要、技能和平台路由中的落地形态。

这份 source 对 wiki 的长期价值在于，它已经形成可复用的 Agent 记忆分类框架：状态层记忆、知识层记忆、学习层记忆；文件系统与 Markdown 适合人机共创和可审计编辑；图结构适合事实关系、时序和多跳推理；混合检索、RRF/MMR/重排、衰减和分层摘要是跨方案共通机制。

## Key Claims

- explicit: 记忆系统由存储层、处理层、检索层和控制层组成，控制层负责生命周期、冲突解决、资源预算和安全治理。
- explicit: 记忆可按场景分为 semantic、episodic、procedural，也可按时间分为短期记忆和长期记忆。
- explicit: 短期记忆不应只视为聊天记录，而应视为 thread/session 的完整运行状态，包含消息、中间推理产物、工具调用结果、TODO、checkpoint 和 resume 信息。
- explicit: 长期记忆需要持久化、可检索、压缩、筛选和更新，常见实现包括向量数据库和图数据库。
- explicit: 记忆压缩、记忆演化、冲突解决和记忆衰减是 AI Memory 的关键问题。
- explicit: 混合搜索结合向量检索和关键词检索，Rerank 对初步检索结果再次排序，RRF 用于多路检索结果融合。
- explicit: OpenViking 使用 `viking://` 虚拟文件系统、L0/L1/L2 分层上下文、目录递归检索和会话提交后的记忆抽取。
- explicit: memory-lancedb-pro 使用 LLM 驱动的六类提取、向量+BM25 混合检索、重排序和 Weibull 生命周期衰减。
- explicit: Graphiti 支持时序事实、事件溯源、预设/习得本体、增量图构建和语义/关键词/图遍历混合检索。
- explicit: Claude Code 的记忆系统基于文件系统和纯文本，包含 `CLAUDE.md`、skills 和自动记忆目录。
- explicit: MemOS 将长期知识、工具轨迹、任务总结和技能进化统一到记忆操作系统中。
- inferred: Agent 记忆方案可按“状态管理优先、结构化知识优先、学习闭环优先、人机协作优先”四类路线比较。
