## Summary

这份 XMind 总结了面向 agent 的工具设计与迭代方法：先快速构建工具原型并亲自测试，再用评估任务衡量 agent 使用工具的效果，最后通过评估推理、反馈、错误和成本指标持续改进工具名称、命名空间、响应结构、token 效率和工具描述。

## Source Digest

该 source 把工具调用看成一个可迭代的产品和评估问题，而不是简单 API 封装。工作流从快速做出工具原型开始，通过亲自试用和用户反馈建立对真实用例的直觉；随后生成贴近真实场景的评估任务，每个评估提示都配套可验证响应或结果，再用直接调用 LLM API 的方式批量评估 Claude 使用工具的表现。评估不只看最终准确性，还要观察总运行时间、调用次数、token 消耗、工具错误，以及 agent 的推理和反馈，从中定位工具描述、规范和响应结构的粗糙处。

工具设计原则强调“少而清晰”。source 明确指出更多工具并不总是更好，LLM agent 的上下文有限，工具不能按传统 API 思路暴露过细接口。更适合 agent 的工具应有清晰且独特的用途，可以把常被串联的多步任务合并为一次调用，在返回结果中附带高价值上下文，并避免把低层技术标识符暴露给模型。命名空间可帮助 agent 在大量工具之间理解边界，例如按服务或资源分组。

响应设计的核心是 token 效率和决策相关性。对大响应应使用分页、范围选择、过滤、截断和合理默认参数，引导 agent 做多次小而精准的搜索，而不是一次宽泛搜索。工具描述和规范本身也需要提示工程，因为它们会进入 agent 上下文并影响调用策略。整体上，这份材料适合沉淀为“工具即 agent-facing interface”的方法论：工具的成功标准不是 API 灵活性，而是 agent 能否稳定、低成本、少歧义地选对、用对并从返回值继续行动。

## Key Claims

- explicit: 工具调用迭代应先快速构建并测试原型，再基于真实使用场景生成评估任务。
- explicit: 每个评估提示都应配对一个可验证响应或结果。
- explicit: 工具评估除了准确性，还应看总运行时间、调用次数、token 消耗和工具错误。
- explicit: 更多工具不一定带来更好结果；工具过多或功能重叠会分散 agent 注意力。
- explicit: 工具可以整合多个离散操作，或把常串联的多步任务封装成一次调用。
- explicit: 工具返回应提供高价值上下文，避免低层技术标识符，并可支持 concise/detailed 等响应格式选择。
- explicit: 对大上下文工具响应，应采用分页、范围选择、过滤、截断和合理默认值。
- inferred: 面向 agent 的工具设计应以“减少模型决策歧义和上下文浪费”为主要优化目标。
- inferred: 工具描述、命名空间和响应结构都属于 Agent Harness 的一部分，因为它们直接影响 agent 的行为边界和反馈质量。
