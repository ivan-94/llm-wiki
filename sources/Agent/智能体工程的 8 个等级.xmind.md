## Summary

这份 XMind 提出“智能体工程的 8 个等级”成熟度路径：从 Tab 补全和智能体 IDE，到上下文工程、复合工程、MCP 与技能、Harness Engineering、后台智能体，再到自主智能体团队。它强调工程重点从单次代码生成逐步转向上下文密度、经验沉淀、能力扩展、反馈环境、后台探索和多 agent 自协调。

## Source Digest

该 source 用等级模型描述 AI 编程/智能体工程的演进。第 1 和第 2 级合并讲 Tab 补全与智能体 IDE：GitHub Copilot 代表按 Tab 自动补全代码，Cursor 这类 AI IDE 将聊天与代码库连接起来，让跨文件编辑更容易；但天花板是上下文，模型要么没看到正确上下文，要么看到太多无关上下文。

第 3 级是上下文工程，关注每个 token 的信息密度。source 指出小模型、token 消耗大的 MCP 工具、接入几十个工具的智能体，都特别容易受上下文噪声影响；上下文工程的重心从过滤坏上下文，转向让正确上下文在正确时间出现。第 4 级是复合工程，强调“计划、委派、评估、沉淀”的循环：上下文工程改善本次会话，复合工程改善未来每次会话，关键动作是把经验教训写入 CLAUDE.md 或等效规则文件。

第 5 级是 MCP 与技能，解决能力问题而不只是上下文问题；source 特别指出 LLM 越来越多使用 CLI 工具而非 MCP，因为 MCP 工具定义会在每轮注入上下文，带来 token 效率问题。第 6 级 Harness Engineering 则把目标从“模型看到什么”扩展为“构建工具、基础设施和反馈循环”，让智能体在低干预下可靠工作，例如把 DevTools、可观测性和浏览器导航接入运行时，使 agent 能截图、驱动 UI、查日志并验证修复。

第 7 级是后台智能体，认为复杂功能的计划模式会越来越像探索：探查代码库、在 worktree 中做原型、摸清解决方案空间，而且这些探索会越来越多由后台智能体完成。第 8 级是自主智能体团队，智能体之间可以直接协调、认领任务、共享发现、标记依赖和解决冲突，不再依赖单一编排者。整体上，这份模型把 agent 工程成熟度从“局部辅助”推进到“可学习、可反馈、可并行、可自治的工程系统”。

## Key Claims

- explicit: Tab 补全和智能体 IDE 的主要天花板是上下文，模型可能缺少正确上下文或看到太多无关上下文。
- explicit: 上下文工程的核心是提高每个 token 的信息密度。
- explicit: 小模型、token 消耗大的 MCP 和接入大量工具的智能体更需要上下文工程。
- explicit: 复合工程改善未来每一次会话，循环是计划、委派、评估、沉淀。
- explicit: 复合工程的常见沉淀方式是更新 CLAUDE.md 或等效规则文件。
- explicit: 第 5 级解决能力问题，并观察到 LLM 越来越多使用 CLI 工具而非 MCP，原因是 token 效率。
- explicit: Harness Engineering 构建工具、基础设施和反馈循环，让智能体能在更少干预下可靠工作。
- explicit: 后台智能体会把复杂功能的“计划”变成代码库探索、worktree 原型和方案空间摸索。
- explicit: 自主智能体团队要求智能体之间直接协调、认领任务、共享发现、标记依赖并解决冲突。
- inferred: 该等级模型与现有 wiki 中 Agentic Engineering、Agent Harness、Agent Skills 和 Vibe Coding 的成熟度叙事高度重叠，适合后续编译为一条 Agent 工程学习路径。
