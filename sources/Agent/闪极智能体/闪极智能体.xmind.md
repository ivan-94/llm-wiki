## Summary

这份 XMind 是闪极智能体的早期架构图，重点定义旧智能体的问题、新架构分层、眼镜主动/被动消息能力、MessageBus、Agent runtime、能力层、成本控制、运行时选型、工具设计、会话状态和输入输出消息模型。它比里程碑文档更偏架构探索和方案对比。

## Source Digest

Source 首先解释旧智能体的核心缺陷：每个智能体能力固定、只能单轮问答、无法跨搜索/日程/飞书/录音等能力编排，且开发抽象落后，难以接入 Skills、MCP 和第三方服务。新架构的目标是用 Agent Loop、工具调用、MessageBus 和能力层，把眼镜从“调用某个固定智能体”变成“承载一个可扩展 Agent runtime 的交互 Channel”。

基础消息能力被拆成主动消息和被动消息。主动消息来自唤醒词或按键，要求短时间连接、多轮交互、多模态、拍照/短视频、设备控制、Human-in-the-loop 和 WebSocket 会话保持；被动消息来自长连接或轮询，用于通知、任务回调和需要用户确认的危险操作。这个拆分决定了设备交互层不能只是 HTTP 问答转发，还要能处理通知、控制、会话、打断和后续回复。

架构分层包括设备交互层、MessageBus、智能体运行时、LLM 网关、应用层和 Channel。MessageBus 解耦设备与 Agent runtime，承载 inbound、outbound、定时任务、控制指令和第三方 Channel；智能体运行时需要兼容 Legacy Agent、新 Agent 和第三方智能体；应用层把闪极应用从旧智能体剥离，通过 API、MCP、CLI、Skills 或 SubAgent 暴露能力；LLM 网关负责标准 API、调用跟踪、成本统计和多供应商后备。

成本部分给出 Agent Loop 与 Workflow 的取舍：低复杂度任务可用 Workflow 或 one-shot 缓存降低成本，高复杂度任务中 Workflow 成本随复杂度快速膨胀，而 Agent Loop 更适合按推理步数线性扩展。成本控制手段包括请求限额、自定义模型、OAuth 套餐复用、混合架构、Prompt 缓存、渐进式披露、SubAgent、模型路由、上下文控制和工具数量控制。

运行时调研把候选分为 LLM Provider、Agent runtime/framework 和沙箱。LLM 网关候选包括 Kong、Helicone、TensorZero；运行时候选包括 OpenClaw/pi-mono/Nanobot、LangGraph/DeepAgent、agno、AutoGen、CrewAI、Pydantic AI、openai-agents-python 和 Claude Agent SDK。工具设计倾向少而强，内置工具不要超过 8 个，分无沙箱版 Read/Run/WebSearch/MemorySearch/MemorySave 与有沙箱版 Read/Write/Edit/Exec。会话状态覆盖短期聊天历史、主动长期记忆、历史压缩、Skills 索引、Prompt 分层和会话持久化。

输入输出模型通过 MessageBus 发布和消费，消息字段包括 type、category、id、user_id、params、timestamp、session_id、metadata、stream_id 和 context；category 包含 toolcall、toolcall_result、answer_start、answer_stream、answer_complete、toast、ask、cron、device_call 等。Source 的两个后续 sheet 只有占位根节点，说明 Agent Loop/Workflow 对比和 Agent 运行时对比尚未展开。

## Key Claims

- explicit: 旧智能体一次问答只能由一个智能体完成单轮调用，难以实现搜索加日程提醒、录音日报、飞书处理等跨能力编排。
- explicit: 完整 Skills 支持需要操作系统级运行界面、文件系统、Shell、Node/Python、浏览器和沙箱；短期可先挂载少量工具、MCP 和文件系统。
- explicit: 新架构应分离设备交互层、MessageBus、智能体运行时、LLM 网关、应用层和 Channel。
- explicit: MessageBus 负责解耦设备交互层和 Agent runtime，并承载输入、输出、定时任务、控制指令和外部 Channel。
- explicit: 应用层能力应从旧智能体中剥离，通过 API、MCP、CLI、Skills 或 SubAgent 形式提供给 Agent runtime 或第三方智能体。
- explicit: 内置工具应尽量少，不超过 8 个；工具过多会增加成本并影响模型性能。
- explicit: 输入输出消息模型需要覆盖工具调用、工具结果、回答流、轻提示、询问、定时任务和设备调用等事件类型。
- inferred: 这份 source 把“AI 眼镜智能体”建模为消息驱动的运行时系统，而不是单个聊天接口或单个业务智能体。
- inferred: 成本控制的关键不是只换便宜模型，而是通过问题分流、上下文裁剪、工具渐进披露、SubAgent 和模型路由共同降低每次运行的无效上下文与工具负担。
