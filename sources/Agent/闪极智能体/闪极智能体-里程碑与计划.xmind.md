## Summary

这份 XMind 把闪极智能体从愿景、生产化路线、MVP 技术规格到测试场景串成一份产品和架构草案。核心方向是把眼镜语音、多模态、录音、闪记、日程和第三方渠道组合为一个可扩展 Agent runtime，而不是继续维护多个互相隔离的单轮智能体。

## Source Digest

Source 的起点是现有 AI 对话和旧智能体架构的局限：能力固定、难以多轮编排、无法把搜索、日程、录音、闪记、第三方服务串联成复杂任务。闪极智能体的差异化不在于拥有最强通用 Agent，而在于把 AI 眼镜的便捷语音入口、多模态环境捕获、全天录音和闪极核心应用能力组合成一个内置智能体，并进一步反向开放为 Channel、CLI 和 Skills。

愿景部分给出一条 AI 眼镜场景分层：从 L1 简单问答与环境捕获，到 L2 工具调用和任务代理，再到 L3 现实增强、L4 Agent、L5 持续陪伴和 L6 主动 Agent。生产化路线围绕链路优化、安全沙箱、能力接入、第三方渠道、成本控制和交互升级展开，强调主动消息、被动通知、多轮消息、模型/Skill 接入和成本跟踪。

MVP Spec 明确做取舍：先聚焦眼镜语音交互、单用户单会话、有限工具能力、单应答串行和闪极业务耦合；暂不追求完整企业级审计、大规模高并发、复杂 SubAgent、规划模式和完整命令交互。技术栈倾向 TypeScript、bun、Drizzle ORM、SQLite、Docker/dev container、nsjail 和 pi-mono agent core。顶层架构拆成 Agent 服务、Gateway、Agent Loop、沙箱服务、LLM 网关、SessionStorage、Skills/Tool/Model registry、ContextManager、Compaction 和 ResourceLoader。

接口设计以 OpenAI-compatible `/v1/chat/completions` 为最小集成面，同时通过 SSE `custom_event` 承载工具调用、思考、工具结果、结构化输出、中断和设备控制等非标准事件。工作区模型区分宿主视图和用户视图，通过用户目录、内置只读 skills、私有 skills、`USER.md`、`MEMORY.md` 和用户 SQLite 数据库支撑持久化会话。沙箱部分选择 nsjail 风格的轻量多用户命令执行服务，用户级 `NsJailService` 负责队列、运行任务和进程执行。

Agent 运行时层面，source 保留了较完整的数据模型和类边界：上下文由代理身份、工具行为、外部 system、MEMORY/USER 快照、技能索引和环境信息组成；持久化层按 `sessions`、`messages`、`messages_fts`、`schema_version` 组织；`AgentSession` 提供 `prompt`、`subscribe`、`abort`、`dispose`，并用 `isStreaming` 暴露串行运行状态；`ModelRegistry` 只要求 OpenAI Compatible API，按 `large/base/small/tiny` 路由任务规模。Case sheet 则把多模态拍照、拍照/录像控制、闪记、日程、飞书消息/待办/邮件/日报作为验收场景。

## Key Claims

- explicit: 闪极智能体的产品差异化来自眼镜语音和多模态入口、闪极录音/闪记/日程等自有能力，以及在此基础上开放 Skills 和外部 Channel。
- explicit: MVP 先采用单用户单会话、单应答串行、有限工具和闪极业务耦合，以降低语音交互和沙箱安全复杂度。
- explicit: Agent 服务需要 Gateway、Agent Loop、MessageBus、SessionStorage、SkillsRegistry、ToolRegistry、ModelRegistry、ContextManager、Compaction 和 ResourceLoader 等运行时组件。
- explicit: `/v1/chat/completions` 被设计为 OpenAI 兼容入口，但需要通过 SSE 自定义事件表达工具调用、reasoning、工具结果、结构化输出、中断和设备控制。
- explicit: 用户工作区需要同时支持内置只读 skills、用户私有 skills、`USER.md`、`MEMORY.md`、SQLite 会话存储和用户私有 workspace。
- explicit: 会话持久化计划使用独立 SQLite，包含 `sessions`、`messages`、`messages_fts` 和 `schema_version`，并按 turn 边界写入消息。
- inferred: 这份 source 的架构核心是把 AI 眼镜从“设备问答入口”提升为一个带 runtime、message bus、sandbox、memory 和 skills 的 Agent Channel。
- inferred: MVP 的主要风险不在 Agent Loop 本身，而在设备事件语义、OpenAI 兼容接口与自定义事件的边界、用户沙箱隔离、会话压缩和闪极业务能力 CLI 化。
