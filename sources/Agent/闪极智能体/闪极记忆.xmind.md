## Summary

这份 XMind 定义 Loomos Memory 的扩展协议和渠道模型。它把记忆设计成多渠道召回层：对话是动态记忆，闪记、相册、日程、录音是静态记忆；Agent 通过统一 search/readMore/installTools 协议检索、展开和挂载渠道专用工具。

## Source Digest

Source 的核心是把“记忆”从单一聊天历史扩展为跨渠道检索系统。扩展机制包含三类能力：`search(input, filter)` 接收原始用户查询和过滤条件，返回 AI 可读记录；`readMore(referenceId)` 用于按引用读取更深层内容；`installTools()` 允许渠道注册专用工具，例如日程的时间范围查询。

协议层把 filter 和 Record 结构显式化。filter 至少包含 `user_id`、`start_datetime` 和 `end_datetime`，让记忆检索可以限定用户和时间范围。Record 则包含渠道内唯一 `id`、AI 可读 `content`、`channel`、多模态 `url`、`mime`、扩展 `references`、原始结构化 `entity`、`created_at`、`score`、`user_id` 和其他元数据。`references` 是多层索引的关键：第一层可以只返回标题或摘要，Agent 需要更多细节时再调用 `readMore`。

渠道模型分动态记忆和静态记忆。动态记忆主要是对话，可在对话被移出上下文触发压缩时写入，也可以由用户显式请求记忆，例如“记录员”场景；底层可以选择流行记忆框架。静态记忆包括闪记、相册、日程和录音。日程需要按时间范围检索的自定义工具；录音可只向量化标题、描述和总结，转录全文不一定进入向量库。

Source 最后对比 Workflow 与 Agent Loop 在记忆场景中的角色。Workflow 适合固定流程、效率较高，但结果一般；Agent Loop 更适合复杂查询、多轮检索、试错、按需加载资源和非常规查询。对于多渠道记忆，Agent 不需要预先知道如何优化每个查询，而是可以由 LLM 决策调用哪些工具、访问哪些渠道、何时展开引用，以及如何处理结构化和非结构化结果。

## Key Claims

- explicit: 记忆扩展机制应支持 `search(input, filter)`、`readMore(referenceId)` 和 `installTools()` 三类能力。
- explicit: 召回 filter 至少包含 `user_id`、`start_datetime` 和 `end_datetime`，用于限定用户和时间范围。
- explicit: Record 应包含 AI 可读内容、渠道、URL、MIME、references、原始实体、创建时间、score、user_id 和其他元数据。
- explicit: `references` 支持多层索引，例如录音可从标题进入总结再进入详情，日程可从标题/描述进入关联录音或闪记。
- explicit: 动态记忆的对话写入时机包括被移出上下文触发压缩时，以及用户显式请求记忆时。
- explicit: 录音渠道可以只向量化标题、描述和总结，转录内容可以不放入向量存储。
- inferred: 这份 source 的记忆设计更接近“统一检索协议 + 渠道适配器”，而不是把所有用户数据合并成一个单一向量库。
- inferred: Agent Loop 对多渠道记忆的价值在于延迟决策：先粗召回，再按 references 深读，再用渠道工具处理特殊查询。
