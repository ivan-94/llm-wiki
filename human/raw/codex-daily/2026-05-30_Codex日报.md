## 分析与洞察

今天最有解释力的成功模式是父 Agent 把批量任务拆成有明确 write scope 的 worker，再用 raw-diff、git diff check 和 wikilink scan 做回归门禁。这个模式让大量 ingest/backfill 可以并行推进，而且 worker 即使出现 patch 落错 checkout 的问题，也能通过后续验证暴露并撤回。下次类似任务应把“工作区确认、落盘路径确认、最终 clean 状态”放进 worker prompt 和父线程验收，而不是只要求 worker 汇报处理文件。

Source Links 修复暴露的问题更深：早期 ingest 把 source note 当作终点，`Links` 像候选清单而不是知识图谱边。用户指出抽象浅、实体稀疏、忽略 raw 目录结构后，修复方向才从“把纯文本改成 wikilink”升级为“补概念/实体、识别 source cluster、让目录关系进入 compile gate”。如果没有这次纠偏，wiki 会继续堆积 source note，看似 coverage 提升，实际长期复习和回溯仍然困难。

日报 skill 本身也验证了一个边界：事实抽取可以脚本化，但日报正文不能机械生成。`render_daily_note.py` 的失败说明，自动化很容易越过“收集事实”和“做复盘判断”的边界；后续 skill 设计应让脚本稳定输出 parent workflow、group、异常、状态和用户输入，再由 Agent 负责取舍、归纳和写作。

本轮公开 thread 工具不可用，也再次说明日报流程不能依赖单一数据面。公开工具适合作为首选候选发现；当工具不可用或信息不足时，Source Manifest 必须明确降级到本机 timeline，并说明覆盖限制。对日报来说，真实可复核的 parent workflow/group 事实比“看起来更公开”的不完整摘要更重要。

## 启发与下一步

- 建立 parent workflow 审计清单。触发条件是任何父线程派发多个 worker，尤其是并行 worktree 写入；检查项应包括 parent_thread_id、agent_role、final message、产物清单、落盘 diff、验证命令和父线程接管记录。不做这件事，worker 中断或误写 checkout 会被日报/验收误判成完成。
- 将 source link 修复沉淀成专门流程。触发条件是 `Links` 中出现 candidate-only、plain text、no-downstream false negative 或 source-to-source 语义可疑；流程应先补足真实概念/实体，再移动候选并修 wikilink，而不是直接批量替换文本。
- 继续收紧日报事实层。触发条件是日报涉及多 worker、多阶段、上下文压缩或异常 session；`collect_session_facts.py` 应默认给出 parent workflow 视图、完整 user message、压缩 tool 关键参数、状态映射和 `--full` 反查路径。
- 补齐 human / vault sync 边界。触发条件是用户开始在 Obsidian vault 内写长期笔记；默认 publish 应覆盖 `human/` 的阅读需求，但 prune 必须保护 `human/inbox` 和 `human/raw`，避免移动端或手写笔记被误删。
- 后续值得形成的技能包括：`ai-wiki-parent-workflow-audit` 用于审计父子工作流完整性，`ai-wiki-link-repair` 用于专门修 source `Links`，`ai-wiki-human-ingest` 用于 human raw Markdown 稳定进入 `human/sources`。
