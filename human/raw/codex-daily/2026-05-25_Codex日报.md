## 分析与洞察

今天最明显的模式是：用户多次把 Agent 从“文档/代码推断”拉回“真实运行路径”。AI Daily waiting 链路就是典型例子。前端看起来支持 `waiting`，BFF 也透传，服务端接口也能按 status 查；如果停在代码阅读，结论会是“链路没问题”。但真实数据证明问题在默认时间窗口：今天稍后执行的 waiting daily 被“截止当前时刻”过滤掉。这个发现只能通过造数据、打 live API、对比旧新窗口得到。下次遇到“列表查不到某状态”的问题，应该更早构造一条会被边界条件命中的测试数据，而不是先在静态代码里猜过滤层。

第二个张力是“规则边界”和“既有示例”会互相污染。xmind-cli 新规则说 Source Manifest 只属于 PRD/issue/HAT/handoff，但 repo 里旧 docs 和 implementation notes 还保留了相反示范。worker 最初只改了规则文本，父 review 立刻发现 agent docs 自己还带 `## Source Manifest`，这会让下游 Agent 继续照旧模仿。规则变更不是只改一句原则，还要清掉最靠近执行入口的反例；否则 Agent 会从局部示范学习到错误行为。

第三个张力是“自动化已实现”和“自动化已生效”不是一回事。agent-game 的 Codex hook 脚本模拟测试通过，但真实会话没有生成 `.codex/state/game_hooks_state.json`，说明 repo-local hook 是否被加载/trust 仍未确认。如果把脚本测试通过当成产品可用，就会高估可靠性。以后凡是 hooks、runtime、scheduler 这类生命周期自动化，都需要分成两层验收：脚本/单元逻辑是否正确，以及宿主程序是否真的加载并触发。

第四个有效做法是先用 grill-me 把窄边界问清楚，再交给 TDD 或 worker。ai_glass 周/月报关闭本来可以扩大成写路径、存量任务、读路径、retry、regenerate 全链路拦截，但用户明确选择“只拦 planner”。这个边界让实现变成一个 scheduler 常量和两个测试分支，并避免了不必要的数据迁移和行为变更。相反，如果 Agent 按“更完整”的直觉实现所有写路径拦截，就会违背用户的止血目标。

第五个风险来自多 agent 并发工作中的父侧集成。xmind-cli 的 slice worker 各自完成得很快，但父侧仍需要处理远端 master 更新、worker 间文档口径重叠、cross-review P2、HAT guide 和 PR body。今天的成功不在于并发本身，而在于父 Agent 没有直接相信 worker 输出，而是重新跑质量门禁、读 review、补集成修复并最终产出 HAT-ready PR。

## 启发与下一步

- 对状态列表类问题，优先造一条边界测试数据。触发条件是“代码看起来支持，但 UI 查不到”。行动方式是用唯一 `user_id` 或标记造最小数据，分别打原始服务、BFF、前端等价 query；这样能快速区分“不支持”和“默认筛选误杀”。
- 对 repo 规则变更，必须检查最近执行入口和示例。触发条件是修改 AGENTS、issue tracker、workflow docs、README 口径。行动方式是 grep 旧规则词、删除或标注反例，并让父 review 专门检查“新规则是否被旧示范抵消”。
- 对 hooks/runtime 自动化，验收要分两阶段。触发条件是新增 hook、scheduler、background worker 或自动续写。行动方式是先跑脚本级模拟，再用宿主工具的 trust/loading/status UI 或日志确认真实生命周期触发。
- 多 slice 交付继续保留父侧集成责任。触发条件是多个 worker 修改同一产品表面或共享文档口径。行动方式是 worker 只提交窄 slice，父 Agent 负责最新基线、cross-review、质量门禁、HAT 和 PR 说明。
