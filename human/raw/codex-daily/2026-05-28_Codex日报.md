## 分析与洞察

今天最明显的张力是“工具可读”和“真实用户可用”之间的差距。`xmind-cli validate` 能证明文件符合本工具解析器预期，但用户直接打开失败后才暴露 XMind App 对真实包结构的要求；后来改用真实 App fixture 才把问题闭环。同样，`ai_glass` 单测能通过，但真实 MySQL/Redis HAT 暴露了 flush 时机问题。下次遇到文件格式、缓存、外部 App 或端到端接口时，不能只用本工具自证；至少要加一个真实消费者路径。

第二个模式是用户纠偏经常不是“再调一点”，而是在纠正任务边界。mindmap 中用户不断把实现从“近似好看”拉回“必须是这样的效果”、从工具栏保留拉回“完全移除”、从隐藏折叠入口拉回“移动端必须容易点”。图片任务里用户明确要求“必须全部 imagegen”，否定了本地重绘这种工程上更可控但不符合任务约束的路径。TTS 任务里用户也纠正了“只保留对白行”导致的过度删除。Agent 下次应更早把“允许的实现路径”和“不能牺牲的内容边界”显式写成验收条件。

第三个值得保留的做法是真实验收中的 seed 数据设计。AI Daily stats HAT 不是只看接口有无字段，而是设计了 miss、hit、refresh、delete、isolation 五类用户数据，并同时抓 API response、DB rows、Redis value。这种 seed 矩阵让一个隐藏事务问题浮出水面。以后做读模型、缓存和统计接口时，HAT 的价值就在于构造能区分“看起来能用”和“状态转移正确”的数据。

第四个风险是生成式图像任务的可控性。TS58 和转速计两批图都说明，imagegen 可以快速生成本地化素材，但对品牌、屏幕、规格、机身细节的保持不是严格保证。工程上本地覆盖更稳定，任务要求上却可能必须 imagegen。下次要先问清楚：用户要的是“全部由 imagegen 生成”的过程合规，还是“商品像素不变”的质量合规；两者冲突时不能默认用工程替代路径。
