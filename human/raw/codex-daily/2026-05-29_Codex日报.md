## 分析与洞察

今天最强的模式是“规则边界必须先收敛，吞吐才有意义”。无论是把 wiki 从 iCloud 直写迁到 git repo，还是把 Image/Markdown/Excalidraw 规则下沉到 skill，本质都是避免多个文档同时描述同一行为。早期 agent 能快速写出规则和脚本，但也会把自己的工程判断混进用户需求，例如 PDF preview `partial` 语义；后续 cross-review 和用户纠偏把它拉回“先保持原需求，再单独讨论设计升级”。以后凡是要改 AGENTS/template/skill 三层文本，应先判断这是全局契约、类型专属规则，还是 helper 实现细节。

第二个模式是“真实产品路径会暴露抽象计划看不到的问题”。Vault sync 的错误不是脚本不能复制文件，而是目标路径从普通 iCloud Drive、大小写目录一路修正到 Obsidian 官方 iCloud 容器。类似地，XMind Markdown export 的问题不是 AI Wiki 规则写得不够详细，而是底层 CLI 导出遗漏了 hyperlink。没有实际 apply 到 iOS 可见路径、没有真实 XMind fixture 验证，规则层会看起来正确但事实链断掉。以后涉及外部 app 或 raw 工具链时，计划阶段就要包含一次真实路径 smoke test。

第三个模式是“父 agent 编排 worker 时，越要克制 worker 的写入范围”。`生图` 和 `Vibe` 两次并行 ingest 都成功，是因为 worker 只写 source notes，compile、index、log、页面创建由父 agent统一处理。这样牺牲了 worker 的局部完整性，但换来低冲突、可审计和更一致的 compile 判断。反事实风险是，如果每个 worker 都创建概念页和修改 index，最后会产生重复概念、弱 evidence 和大量 merge 冲突。

第四个模式是“工具能力和 wiki 方法论互相推动”。AI Wiki 需要可靠读 XMind，于是反向推动 `xmind-cli` 修 iCloud placeholder、Markdown hyperlink、patch 安全写入；`xmind-cli` 的能力增强又让 AI Wiki source note 更完整。这个闭环说明个人知识库不是单独的 Markdown 项目，而是 raw 工具链、agent workflow、同步路径、验证脚本共同组成的系统。

## 启发与下一步

- 把“大批量 ingest”固定成触发式流程：raw diff preflight -> 父 agent 隔离 worktree -> worker 只写 source notes -> 父 agent merge/compile -> raw diff 清零。触发条件是同一目录超过 10 个 supported raw，或需要并发 worker。
- 涉及 Obsidian/iCloud/macOS 文件提供器时，必须先定位真实容器并做 dry-run/apply smoke test。触发条件是路径里出现 `Mobile Documents`、iOS 查看、CloudDocs、`iCloud~md~obsidian` 或 placeholder 文件。
- 修改规则文本前，先做“规则归属判定”：AGENTS 只放全局硬规则和路由，类型细则进 skill，具体命令行为进 helper/test。触发条件是同一句规则可能同时出现在 AGENTS、template、skill 或脚本注释里。
- 对底层工具链问题不要只在 wiki 层 workaround。触发条件是 source note 缺证据来自工具导出缺字段、只读单 sheet、iCloud placeholder 或格式解析问题；优先修工具并加 fixture 测试。
