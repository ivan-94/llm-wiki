## 分析与洞察

今天最明显的模式是，真正推动工作前进的不是“再开一个新方向”，而是先把已有方向拆成更小、更可验证的 contract。OSS 迁移先做四路只读探索，再限定写作用域；Quick Note 先补后端匹配 contract，再让前端去接；`arms-exceptions` 则不是继续加 skill，而是反过来压缩已有 skill 主文件。这个模式的共同收益是，后续每一轮实现都更像在填具体缺口，而不是重新定义任务边界。

第二个洞察是，今天多条线都在证明“活跃线程并不一定是坏事，但前提是主线程已经先收敛了风险形状”。OSS 迁移的两个子线程还活跃着，但父线程已经完成探索、helper 落地、独立 review 和一轮签名参数纠错，所以当前的未完成更像是尾部收口，而不是方向不清。相反，如果没有前面的探索和 review，这两个活跃线程就会变成不透明的风险。

第三个洞察是，DSL rewrite 的难点已经不再是语法，而是信息分层。今天多份 `arms-exceptions` skill 都能机械过校验，说明“写成 DSL”这件事本身已不难；反复重写同一 skill 则说明真正困难的是决定哪些内容算 contract，哪些只是手册。后续如果不把这层分界继续标准化，重复 rewrite 会持续消耗时间，却不一定继续提高可执行性。

## 对用户/Codex 交互的批评

今天未见显著的用户侧交互缺陷。多数任务都先给了明确边界，例如“只读探索”“限定写作用域”“最多 1000”“不要改旧接口语义”，这直接降低了 Codex 擅自扩范围的空间。尤其 OSS 迁移那条线，先探索再实现的互动方式明显优于直接丢一个大改造请求。

Codex 侧仍有一个值得警惕的倾向：在 DSL rewrite 这类“看起来很熟”的任务里，容易把多次相近请求都当成独立新活来完整重写一遍。虽然最终结果都过了机械校验，但从当日 session 分布看，`arms-exceptions-triage` 的重复 rewrite 暗示 Codex 还不够主动地区分“这是新边界”还是“这是同一边界的重复触发”。更好的做法是更早检查当前文件是否已经满足新请求，避免把重复劳动包装成新进展。

另一个值得保留的互动模式是今天多次使用“探索子任务 + 主线程汇总 + 独立只读 review”。这在 OSS 迁移里效果很好，因为它把真实改动前的未知数压缩掉了。但这类模式只有在父线程明确汇总和裁决时才有价值；如果只是多开线程而不做最终聚合，成本会迅速反噬。

## 启发与下一步

1. 对跨文件迁移任务默认采用“探索分片 -> 限定写作用域 -> 独立 review -> 父线程纠错”的节奏。触发条件是像 OSS 迁移这种接口面分散、兼容风险高的任务；不这么做，最容易重复今天自审指出的默认字段外溢或签名参数非法问题。
2. 对 DSL rewrite 类任务增加“是否已满足当前请求”的前置检查。触发条件是同一天或同一文件被重复要求 rewrite；不这么做，会继续出现今天这种同一 skill 多次重写但信息增量有限的情况。
3. 把 Quick Note 这类前后端联动任务拆成“后端 contract 完成”和“前端消费完成”两个显式里程碑。触发条件是后端先补字段、前端后续接入；不这么做，容易把“接口已补齐”误判成“功能已真正完成”。
