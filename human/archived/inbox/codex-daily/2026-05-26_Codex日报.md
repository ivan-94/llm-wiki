## 分析与洞察

今天最强的模式是“规格看起来已经足够，真实链路仍然会推翻一部分设计”。PRD 37 在 issue 阶段已经把 Quick Note 和 Voicemaster 切成了看似清楚的 slices，但 HAT 和用户现场操作连续暴露了三类问题：外部 ASR 503 让成功路径无法验收，Quick Note 表格信息量过载，双重重试按钮对用户语义不清。这里的机制是，代码完成只能证明实现了入口，不能证明运维用户能理解并稳定使用。以后涉及后台管理界面时，应把“真实页面密度、按钮语义、失败路径可解释性”纳入 P0/P1 验收，而不是等 HAT 后再补。

第二个张力点是“保持业务逻辑一致”和“给管理后台开后门”的边界。Voicemaster admin retry 最初讨论了不扣额度、强制重试、状态限制、旧 callback 竞态，最后收敛成薄封装现有 `start_offline_task`，只用 `consume_quota=False` 扩展，priority 固定。这是正确方向：管理后台可以有运维能力，但不该复制一套新转写业务流。否则后续用户端和后台端会在 quota、priority、callback、status 更新上逐渐分叉，bug 会变成“同一个 message 两种世界线”。

第三个信号来自图片本地化：自动化验证和视觉合格不是同一个概念。日语和法语任务都能让 `verify_outputs.py` 通过尺寸和元数据检查，但图像质量问题包括文字残留、白色覆盖块、屏幕读数漂移、画幅重排甚至完全无关生成。这个任务说明，对于视觉资产，脚本验证只能做技术门槛；最终合格必须有人工或视觉模型逐张检查，且要把“不采用不合格候选”作为产物质量规则，而不是为了完成数量而覆盖最终目录。

第四个模式是 `agent-game` 的文件系统重构反复验证了一个架构原则：大世界可以复杂，运行面不能复杂。旧结构的问题不是内容太多，而是当前事实散落在太多位置，导致每次接手都要先判案。用户持续纠偏“不要用旧约定否定式描述当前规则”“index 只做索引”“地图要表达真实空间关系”，说明 Agent-friendly 文档应该更少、更正向、更接近操作入口。以后做重构文档时，少写历史包袱，多写当前唯一事实来源。

第五个有效做法是 parent workflow 编排中的“父级 review 后再合并”。#39 的重复 object key、#40 的取消确认 unhandled rejection、媒体 URL 鉴权风险，都不是 worker 自报中优先暴露的问题，而是父级 review 和独立 review 抓出来的。这说明多 agent 并行的收益不只在速度，也在父 Agent 保留集成责任：看边界、看依赖、看 HAT 可验性、看用户操作路径。
