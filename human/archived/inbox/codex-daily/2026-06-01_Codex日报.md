## 分析与洞察

今天最关键的机制变化是“把验证前移到真实路径”。无论是 cook-tweet 的 X 可见性波动，还是 ai_glass 的 HTTP/Celery 异常捕获，纸面规则都不足以保证可执行性。只有在真实路径里触发失败，才能准确定位 contract 该加在什么层级：是 skill 的 fail-fast/reopen，还是 observability 的 middleware/signal 边界。

[^1]第二个洞察是子 Agent 边界必须按执行现实设计，而不是按理想职责分工设计。cook-github 最初让 low-effort 子 agent 直接落盘报告，看似清晰，实操却被 sandbox 权限击穿。改成“子 agent 返回正文、父 agent 落盘”后，责任更集中，也减少了跨 sandbox 依赖。这说明在多 agent 流程里，最终写入权应优先收敛到父 agent。

第三个洞察是“流程完整”不等于“证据充分”。ai_glass 的 cross-review 两次都显示“有结论无细节”，形式上完成调用，但不能支撑工程决策。相比之下，HAT 的 console span 和请求/task 实证更有决策价值。后续需要把“输出结构有效性”当作流程成功条件，而不是只看命令执行成功。

## 启发与下一步

1. 为跨 agent 任务建立默认写入策略：子 agent 只产出结构化正文，父 agent 负责持久化和最终文件一致性检查。触发条件是任何涉及 `cache/` 或 wiki 持久产物写入的任务。
2. 为 cross-review 增加“结果结构门禁”：若外部轨没有 file:line + 风险 + 建议，立即判失败并自动触发一次受控重试。触发条件是外部 reviewer 输出仅摘要或统计。
3. 对“真实路径优先”的验收方式做模板化：在 skill 验收和可观测性变更中默认要求至少一个真实请求/真实任务证据，再接受测试结论。触发条件是涉及外部平台行为或 runtime 路径的变更。
