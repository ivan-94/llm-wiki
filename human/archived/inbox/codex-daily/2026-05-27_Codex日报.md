## 分析与洞察

今天最明显的模式是：业务代码本身往往能较快收敛，真正反复出问题的是“并行交付的共享事实层”。PRD #71 的 #73/#75 都不是功能逻辑失败，而是 shared `implementation-notes.html` 基于旧父分支修改，合并时会回退 #74 的状态和 Source Manifest。类似地，PRD #82 的 media preview 不是 SQL 写不出来，而是 preview 与 `quick_note_cursor` 是否共享同一个增量窗口这个交接语义没有被原始实现完整带入。以后父 Agent 审查不能只看业务 diff，还要把共享 notes、cursor/window、HAT guide 入口视为一等交付物。

第二个张力点是真实 HAT 多次推翻“单元测试已经覆盖”的安全感。PRD #77 的 HAT 首轮失败是 Open Platform 缺 Bearer token seed，PRD #78 首轮失败是旧模板 override，PRD #71 真实链路暴露 Milvus stale-vector 污染，PRD #82 guide 里路由写错，PRD #81 HAT prepare 又被用户指出必须补真实日记生成。单测负责缩小行为边界，但真实验收负责发现环境、认证、路由、旧数据和外部服务假设。下次 HAT prepare 应默认包含至少一条“真实入口 + 真实 runtime + 真实数据”的正向路径，不能只准备单元测试或局部 helper。

第三个有效做法是用户多次用 grill-me 把需求从“看似简单字段”收窄成可实现契约。Quick Note 年月筛选从新增接口改成扩展原接口，AI Daily duration 从“可从 history 推导”收窄成只用顶层/overview，AI Daily 无录音硬条件明确 `ids` 才是存在信号，Home State preview 明确 image/video、列表形态、cursor 同步。这个过程看起来慢，但它减少了实现后返工的成本；真正返工的地方，恰恰是没有在最早 grill 阶段问到的 cursor 同步语义。

第四个教训是工具链发布需要真实消费者路径收口。`xmind-cli` Homebrew 链路如果停在配置或 PR 合并，就会漏掉 strict audit 的 `version` 问题和 PATH shadow 现实问题。当天最终通过 GitHub Release、tap formula、`brew install`、`brew test`、`brew audit --strict --online` 才证明“用户能用 Homebrew 安装”真的成立。类似标准应该迁移到服务端 PR：只要需求是用户/客户端可见，就要至少有一次真实入口结果展示。
