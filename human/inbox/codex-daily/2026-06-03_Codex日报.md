## 分析与洞察

今天最清晰的规律是，复杂工作流一旦真的要交给下游 Agent 复用，就不能只写“会做什么”，必须写清“谁做、做到哪一层、如何验”。`arms-exceptions-triage` 最开始很容易滑向一个无边界的大父线程，但用户一句“你不要替 sub agent 干活”把结构拉回正轨，后面的真实 triage/fix 闭环才成立。这个纠偏说明，多智能体流程的核心不是多派几个 worker，而是把父线程限制在调度、聚合和裁决。

第二个洞察是，真实安装和真实验收比文档 review 更能暴露 contract 缝隙。今天连续撞到的 `--json` 位置不兼容、`~/.zshrc` 新环境变量不自动继承、`trace_console_url` 不能靠猜、飞书 card 可读性不够，都是“看起来小、但会让 Agent 日常卡住”的问题。它们的共同点是，不跑真实命令就看不出来；一旦跑了，修复也都很具体。这说明后续对面向 Agent 的 CLI/skill，最有价值的不是再写更多说明，而是尽早让真实安装与真实发送跑起来。

第三个洞察是，今天的成功并不是把 scope 做大，而是反复把 scope 收窄。`ai_glass` 的日志治理如果不先限定 P0、线上路径、不改既有 logger 业务语义，就会很快变成“为清理而清理”；`arms-exceptions` 的通知能力如果不强制通用 skill 不耦合 ARMS 字段，也会把基础能力写死。今天多次用户纠偏都在做同一件事：把抽象层次拉回正确位置，让每层只承担自己的责任。

## 对用户/Codex 交互的批评

今天最明显的问题主要在 Codex 侧，而不是用户侧。Codex 有几次暴露出典型的“先替系统假设正确答案，再往下实现”的倾向，比如父 Agent 一度下场干 subagent 的诊断活、`lark-notify` 一度准备硬编码 ARMS 专属 label、`trace_console_url` 一度想先写成要求再说。这些都不是结果导向的高效行为，因为它们会把后续真实运行中的返工成本转嫁给用户。更好的方式是先把通用层和专用层切开，再让真实 explorer/CLI 产出证据，其他组件只消费证据。

用户侧的交互方式整体是有效的，但成本在于需要高频、短促地做边界纠偏，例如“不要硬编码”“不要替 sub agent 干活”“不要硬拍，先验证”。这种风格能快速把方向拉正，但如果系统长期依赖这种人工纠偏，说明上游 contract 还不够硬。更好的协作方式是让技能文件和 workflow 本身把这些边界写成默认行为，下次遇到类似任务时不用再靠聊天里临时刹车。

另一个值得点名的是“真实使用触发修正”这件事今天非常高价值。CLI `--json` 兼容问题、Yunxiao token 环境继承问题，都是用户实际运行命令后才暴露出来的。如果只是停留在代码自测，这些问题会带着“已完成”的假象进入下一轮。这个互动模式本身是对的，后续应保留，但应尽量把这类真实 smoke 提前到实现完成后的第一时间，而不是等用户碰到。

## 启发与下一步

1. 对所有多智能体 workflow 默认补“父 Agent 禁止下场业务探索”的硬规则。触发条件是存在 diagnose/fix 类 subagent 时；不这么做，最容易重复今天这种边界滑动，再靠用户口头拉回。
2. 对所有 Agent-facing CLI 默认加“真实安装 / 真实 config / 真实 send”三类 smoke。触发条件是新增命令、调整帮助信息、改配置契约或加通知能力时；不这么做，会重复今天 `--json`、环境变量、链接格式这类只有真实运行才会暴露的问题。
3. 对 triage/fix 工作流持续强化通用层与专用层分离。触发条件是基础通知、格式化、链接生成、环境 doctor 开始夹带业务字段时；不做这层分离，后面每加一个新业务都要重写通用设施。
