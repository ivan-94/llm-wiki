## 分析与洞察

今天最有价值的一点，是“先写 skill，再跑真实项目”并没有变成纯方法论自嗨，反而让 skill 和项目互相校验。`migrate-frontend-i18n` 在 `ai/skills` 里收敛出来的规划文件、角色分工和验收门禁，确实足够支撑 `web-app` 的多切片实施；但同一天的真实交付也证明，只有真正跑到 runtime、浏览器和共享组件层，才能知道 skill 里哪些约束还只是纸面完整。

第二个洞察是，前端 i18n 的难点不在“替换多少中文”，而在“如何控制共享依赖和验证边界”。今天的 worker 切片本身都能完成自己的模块迁移，但真正的风险出现在跨切片集成处：共享组件从别的路由进入时带来的 namespace 依赖、ready gate 带来的瞬时 missing key、验收脚本没有把 console missing key 当失败。这些问题如果没有父线程集中验收，很容易在“每个 slice 各自通过”时被漏掉。

第三个洞察是，用户持续压缩 skill 规则这一点是有效的，但压缩不能只靠删字数。今天 skill 的改进路径不是“把所有规则展开”，而是把真正会影响执行质量的几个门槛保留下来：基线冻结、planning file、文件级 TODO、slice 门禁和验收证据。换句话说，通用性不是少写约束，而是只写那些跨项目也仍然成立、能挡住偷懒和误判的约束。

## 对用户/Codex 交互的批评

今天的用户侧交互总体是高质量的。用户没有只停留在“做个 i18n skill”或“做个 i18n 迁移”，而是不断把问题压到真正会影响执行的地方，比如不要硬编码太多规则、规划文件必须有模板、按需加载和语言切换入口要对齐、最后还专门追问 skill 在真实执行里暴露了什么问题。这种追问方式显著减少了“看起来完整但落不了地”的风险。

Codex 侧更容易暴露的问题，是前期仍然偏向阶段式、显式 gate 过多，哪怕用户已经给了很强的授权和方向。这在 shared skill 设计阶段还可接受，但到了真实交付里，如果还是机械地把每个阶段都当成必须停下来问人的闸门，就会拖慢节奏，也会让“详细规划”和“开始实施”之间出现不必要的重复确认。更好的方式是把“何时必须停下来问”写成明确条件，而不是默认每阶段都中断。

另一个值得批评的点，是 Codex 在多子线程执行时，天然容易把“每个 worker 都完成了”当作任务已完成。今天真正保证质量的不是 worker 成功，而是父线程把 runtime missing key、acceptance runner、共享 namespace 依赖和收尾文档补齐。下次如果只盯子线程完成率，不把集成和验收单独当作一条工作流，仍然会留下隐患。

3. 把 integration fix 的边界写清楚。触发条件是多 slice 并行实施后出现共享 infra、共享组件或全局检查项问题；如果不写清楚，manager 不是过度接管，就是没人负责最后一公里。
