## 分析与洞察

今天最强的一条共性，是多个线程都在把“靠经验判断”的工作压成能复跑的制度。`ai-wiki-human-raw-diff` 把 human raw 同步从“人工猜哪些变了”变成了有桶、有元数据比较、有误报修复的工具；`codex-session-manager` 把会话整理从想法变成了真实线程工具边界、dry run 契约和自动刷新规则；`ai_glass` 则把模板开发从单次还原推进到 fork、preview、visual diff 和 Runtime HAT 的稳定流程。它们背后是同一个机制：今天有效的进展都不是直接多做一点，而是先把规则写成可以被下次重复执行的最小系统。

第二个洞察是，视觉与内容匹配这类问题，不能靠代码结构反推，只能回到视觉稿。模板区分度那条线之所以最后能通过真实 HAT，不是因为多加了几个标签，而是先承认“所有模板都支持图片，所以图片数量不是关键维度”，再回头从视觉稿里读出正式程度、情绪强度、叙事形态和读者预期。这一点也解释了为什么 `red_summary_scrapbook_01` 最后能在“生活 + 工作 + 雨夜散步”的混合日里胜出：它不是最漂亮的图，而是最能承载“有行动项但不完全冷硬”的那种语气。

第三个洞察来自 dry run 和多 worker 线程。今天两次 `codex-session-manager` dry run 都说明，如果不给 agent 一个明确的收口条件，它在最后报告前会犹豫；而在模板 fork 任务里，恰恰是因为主线程把写入范围完全拆开、只保留统一 review/验收，三路 worker 才没有互相污染。这说明并发工作的核心不是“多开几个 agent”，而是先把写边界、事实来源和主线程收口责任说死。

## 对用户/Codex 交互的批评

今天用户与 Codex 的协作总体质量很高，尤其在 `ai_glass` 模板和会话管理这两条线里，用户都能在关键节点把错误方向拉回来，例如明确“不要把模板选择建立在图片数量上，而要回到视觉稿本身”，以及明确“这个 skill 未来会放进定时器，所以不是副驾驶，而是自动执行器”。这类纠偏很有价值，因为它们直接改变了设计重心，而不是只改一个局部答案。

Codex 侧的主要问题仍然是会在“看起来合理的工程抽象”里走得太快。最明显的两个例子，一个是先把模板选择归因到功能能力和图片数量，而不是先看视觉稿；另一个是三套新模板背景初版直接把 Figma 手机容器圆角烘进了图片里，直到用户在 preview 里指出来才回头修。这两处都说明 Codex 在面对视觉或产品判断时，还是容易先从结构和实现偷懒，而不是先从真实表面现象出发。

另外，`codex-session-manager` 的 dry run 停在最终报告前，也暴露了 Codex 的另一个惯性：它在信息已经足够时仍然可能继续等待一个“更完整”的结论，而不是按规则及时收口。对自动化 skill 来说，这不是小问题，因为它会直接把定时任务拖成悬挂线程。

## 启发与下一步

1. 对视觉模板、内容选择和风格匹配问题，优先读取视觉稿，再谈代码抽象。触发信号是用户在问“这个模板适合什么内容/心情”，而不是“这个组件支持什么字段”；如果不这么做，最容易把真正决定体验的东西错归到代码能力上。
2. 对会进 automation 的 skill，必须显式写出 dry run 和收口规则。触发信号是任务开始涉及 `list_*`、批量建议或状态报告；如果没有强制结束条件，agent 很容易在最后一步继续等、不敢停。
3. 对可并行的同构实现任务，继续坚持“worker 只写自己的目录，主线程负责统一验收”的模式。触发信号是多个 slice 结构相同但资产不同；不这么做，最容易出现目录互改、验证口径不一致和主线程无法快速判断哪一份真的可交付。
