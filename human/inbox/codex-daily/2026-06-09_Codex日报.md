## 分析与洞察

今天最重要的模式是，任务一旦从“需求/结构”跨进“真实页面/真实 artifact”，验证方式就必须立刻改变。`ai_glass` 的时间轴正文需求之所以推进得稳，不是因为 prompt 讨论得多，而是因为它很快被收敛成一个 issue、一个 HAT、一个真实 `diary.html` artifact；`ai/game` 的 `view.ejs` 改造之所以能持续推进，也是因为它没有停留在架构口头描述，而是反复回到真实浏览器页面、截图和可见 DOM 效果。反过来看，`ai_glass` 的视觉模板线之所以痛苦，恰恰是因为文本对齐已经把人带进一种“需求够清楚了”的错觉，但真正决定结果的是默认样式、素材缓存、绝对定位和资源别名这种只能在真实渲染中暴露的问题。对 Codex 来说，这意味着进入高保真界面工作时，截图和运行时检查不是收尾，而是中前期工具。

第二个洞察是，“把边界做窄”仍然是今天最有效的推进方式。时间轴日记需求最后没有拆成多个 issue，而是收敛成一个完整 slice；`view.ejs` 线程里最有效的决策也都在缩小复杂度，比如复用现有 `instance.page`、主服务单 ws、任务中只读、同一角色一次只允许一个动作。相比之下，任何把职责散开或把未验证细节提前耦合在一起的做法，都会制造后续返工。这个模式和昨天的接口收口类问题其实一致，只是今天它在 UI 和交互领域再次出现。

第三个洞察来自 cook-github 这条线。它说明 AI Wiki 里的“知识沉淀”不再只是简单摘要，而是在验证一种工作流：主线程维护 provenance 和写作边界，子线程做严格只读探索，最后产出既能给人读、又能给后续 agent 重放来源的 inbox 笔记。和代码任务相比，这条线的价值不在于技术难度，而在于它把“多智能体协作也需要像工程一样保留来源与边界”这件事做成了可复用模板。

## 对用户/Codex 交互的批评

今天最明显的交互成本来自 `ai_glass` 的视觉模板线。用户在这个方向上给了很多即时纠偏，这本身是必要的，但如果纠偏总是在“已经做出不少实现”之后才通过截图逐条指出，就会导致大量工作看上去像在前进，实际只是把真正的问题向后拖。更好的协作方式是更早把视觉验收标准显式化，例如一开始就给出“先只对齐 5 个装饰锚点、2 个文案锚点、1 张页面全图”的最小验收面，而不是让模板先铺开再逐项回修。否则 Codex 会天然偏向先把看得见的代码做完，再被真实页面打回来。

Codex 侧今天也暴露了一个典型倾向：遇到复杂前端/模板任务时，很容易把“继续修”理解成继续改 CSS/坐标，而不是先重新确认当前页面到底是不是目标实例、当前预览到底有没有吃到新资源、当前偏差究竟来自设计理解还是浏览器默认行为。`figure` 默认 margin、manifest 缓存、SVG 别名这类问题都说明，如果没有先确认运行时事实，后面的每次“继续修”都可能是在错误前提上加工作量。

`ai/game` 与 cook-github 这两条线的交互相对更稳，因为用户给出的目标边界更像工程 slice：一个是“取代 proxy-engine 的交互 view”，一个是“只读 cook 一个指定仓库”。它们的问题不是沟通模糊，而是需要 Codex 主动把隐含复杂度拆出来并及时用真实验证回压。

2. 当一个需求既改数据边界又改渲染行为时，优先把它收敛成可验收 slice，例如 issue + HAT + artifact。触发信号是用户开始同时讨论 prompt、模板、结构化字段和 UI；这样可以避免讨论很多但迟迟没有一个真实交付物来压实边界。
3. 对浏览器页面类 bug，先确认“当前页是否真是目标实例、目标资源是否真已加载”。触发信号是用户说“继续修”，但反馈始终停留在视觉错位或资源不生效；要改变的是把运行时确认放在样式修改之前，不然最容易重复今天的缓存/默认样式类返工。
