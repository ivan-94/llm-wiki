# 异常治理闭环

## Definition

异常治理闭环是从生产异常到 Agent 可消费的结构化任务、自动生成修复 PR、验证、发布、关闭异常的端到端流程；其关键能力不是日志量，而是能否把异常聚合成带代码上下文、影响面和生命周期状态的可分派任务对象。

## Why It Matters

Agent 不擅长从海量非结构化日志中拼上下文，但擅长基于结构化任务做代码修改。如果异常平台能把生产错误聚合为结构化 Issue，Agent 就可以直接消费事故现场并生成修复方案。

## Mental Model

异常治理 = 生产报警 → 结构化 Issue → Agent 可消费上下文 → 修复 PR → 验证 → 关闭。

## Closed Loop Steps

1. 生产异常触发：App/前端/服务产生 error，被监控捕获。
2. 结构化聚合：异常平台对同类 error 去重、聚合为单个 Issue，附带 stack trace、source file/line、release、transaction、request context、breadcrumbs、suspect commit。
3. Agent 可消费：Issue 结构化到足以作为 Agent 任务输入，不需要人工拼凑上下文。
4. 修复任务分派：自动或人工触发 Agent 读取 Issue、分析代码、生成修复 PR。
5. 验证发布：CI/CD 验证，人工审核，上线。
6. 关闭或回归：发布后关闭 Issue；若修复引入新异常，自动重开或触发新 Issue。

## Agent-Consumable Context

理想的异常 Issue 应包含：
- `error_type`、`message`、`stack_trace`
- `source_file` + `line_number`（带 commit hash）
- `environment`、`release`
- `transaction`、`request_context`
- `tags`、`user_context`（影响面）
- `breadcrumbs`（操作序列）
- `event_frequency`、`affected_users`
- `suspect_commit`、`linked_github_issue`

## Key Claims

- explicit：Sentry 的 grouping/去重能把同类异常聚合为一个 Issue，减少告警噪音。
- explicit：Release + Commit + Suspect Commit + GitHub 集成让异常能进入研发工作流。
- explicit：阿里云 ARMS/SLS 等默认是分散可观测底座，需要额外整合才能形成等同闭环。
- inferred：对 AI 修复工作流，异常平台的关键不是日志量，而是能否把异常归一成带代码上下文、影响面和生命周期状态的任务对象。

## Examples

- Sentry Issue + GitHub integration → Agent 读取 Issue → 生成 fix PR → Sentry 关联 release。
- 动态治理流程：Sentry 异常 → GitHub Issue → Codex/Claude Code 自动修复 → PR review → 上线 → 异常关闭。

## Common Confusions

- 异常治理不等于加更多日志；结构化聚合和生命周期状态比日志量更重要。
- 告警平台和异常治理平台职责不同：前者负责告警路由，后者负责异常对象的生命周期管理。
