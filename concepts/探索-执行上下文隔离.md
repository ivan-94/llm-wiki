# 探索-执行上下文隔离

## Definition

探索-执行上下文隔离是把 Agent 工作流中"探索/研究/设计"阶段和"执行/实现"阶段的上下文彻底分开，让每个执行阶段只加载局部精炼上下文，防止探索阶段积累的大量信息污染或稀释执行阶段的核心约束。

## Why It Matters

当探索阶段的思维过程、研究材料和讨论痕迹留在上下文中，执行阶段会受到噪声干扰：关键约束被稀释、模型注意力分散、产出质量下降。分离后，每个阶段的上下文都更精准，模型在各阶段都能发挥 sweet spot 性能。

## Mental Model

探索 = 宽上下文 + 发散；执行 = 窄上下文 + 聚焦。中间有 handoff 文档作为翻译层。

## Phase Separation

### 探索阶段（Research / Planning）
- 目标：理解代码结构、模块依赖、历史记录、相关文档、潜在方案。
- 上下文：宽泛，包含问题背景、代码库探索、多方案评估。
- 输出：方案文档、接口设计草案、架构决策、Mini Spec 或设计 spec。
- 关键动作：人工在设计文档生成后审查，提供"研究阶段最高杠杆的干预点"。

### Handoff 文档（Bridge）
- 提炼探索结论，剔除探索噪声。
- 明确执行阶段的任务、接口、依赖和验收条件。
- 只保留执行需要的核心信息。

### 执行阶段（Implementation）
- 上下文：从 handoff 文档加载，不带探索历史。
- 任务：I/O 闭合、可独立验证，不需要跨任务推理。
- 分批执行：每次重置上下文，确保执行干净。

## Key Claims

- explicit（实战）：开发流程中应频繁、有意地压缩上下文，按阶段去冗余、分层并聚焦下一步需要的信息。
- explicit（实战）：人工审查在研究和设计阶段杠杆更高，可降低实现后返工成本。
- explicit（Sweet Spot）：子任务应可独立执行、I/O 闭合、不依赖隐式状态、可单独验证。
- inferred：探索-执行隔离是"上下文拥有权"（agent 主动管理上下文）的操作层实现，而非只是被动接受所有信息。

## Examples

- 先用独立会话研究一个模块的架构，输出 design doc；再开新会话，只加载 design doc 和相关文件，执行实现。
- 偏移控制六张图中的"语义图"阶段 = 探索输出；实现阶段只加载语义图 + 当前模块。

## Common Confusions

- 隔离不是永远新开对话；是在每个执行切片开始时，只加载"这个切片需要"的上下文。
- 探索和执行之间不是硬界限；对小任务，它们可以在同一对话中完成，但仍应有意识地切换聚焦方向。
