# 会话分层存储与压缩

## Definition

会话分层存储与压缩是一种把 Agent 会话历史拆成「活跃可恢复层」和「已归档摘要层」的持久化设计：用类 Git 的工作区/暂存区/仓库分层，配合保护头尾和增量摘要，解决长会话的断点恢复、上下文压缩和重复摘要问题。

## Why It Matters

长会话必然会超出上下文窗口。粗暴截断会丢上下文，反复全量摘要会重复劳动并漂移。分层存储让「可恢复的活跃上下文」和「可归档的历史摘要」分离，既能断点续跑，又能压缩而不重复提交，是长期 Agent runtime 的基础能力。

## Mental Model

借用 Git：运行时会话历史是工作区，`messages_staged` 是暂存区，`messages` 是仓库。每个 turn 把工作区落到暂存区；压缩时把暂存区里「该归档的部分」提交进仓库，并重置暂存区。

## Key Claims

- `会话存储与会话压缩.xmind`：`messages` 类比 Git 仓库（已归档历史），`messages_staged` 类比暂存区（活跃未提交历史），运行时会话历史类比工作区；完整历史 = `messages + messages_staged`。
- 恢复会话时直接完整加载 `messages_staged`。
- 触发压缩时，对 `messages_staged` 中除 Protect Tail 外的 Summary Area 做摘要并提交到 `messages`；忽略 Summary Area 中 summary 消息之前的 Protect Head（已在上一轮提交过），避免重复提交。
- 压缩后 `messages_staged` 重置为新的保护头、压缩 summary 和保护尾。
- `闪极智能体 2.excalidraw` 印证同一套关键词：messages、messages_staged、Protect Head、Protect Tail、Summary Area、append/replace/keep。
- Hermes 用 SQLite + FTS5 实现会话血缘（lineage），支持压缩前后的续篇恢复；闪极 MVP 用独立 SQLite 的 `sessions`/`messages`/`messages_fts`/`schema_version` 按 turn 边界写入。

## Examples

- 设备断连后重连，Agent 直接加载 `messages_staged` 续上下文，不需重放全部历史。
- 上下文接近上限时触发压缩，把中段对话摘要进仓库，保留最近若干轮（保护尾）保证连续性。

## Common Confusions

- 这不是「换个数据库存聊天记录」，而是把历史按「活跃/归档」分层并定义压缩提交规则。
- Protect Head 和 Protect Tail 作用不同：尾保护最近上下文连续性，头避免重复提交已归档部分。
- 压缩属于 [[concepts/Agent 记忆生命周期|记忆生命周期]] 的一环，但这里聚焦会话历史的存储结构，而非语义记忆抽取。

## My Understanding

当前理解：会话分层就是「工作区/暂存区/仓库」三层加保护头尾——活跃层负责恢复，归档层负责压缩，保护区负责连续性和不重复提交。
