# 提示词缓存

## Definition

提示词缓存（prompt caching）是 LLM 供应商提供的请求间缓存能力：把重复出现的提示前缀（如长系统提示、文档上下文）缓存起来，后续请求命中缓存即可降低延迟和费用。各供应商在自动/隐式/显式触发方式上差异很大。

## Why It Matters

长系统提示、RAG 上下文、Agent 历史会被反复发送，重复计算昂贵。提示词缓存能显著降本提速，但能力是“API 层差异”而非单一模型技巧，选型时需逐供应商比较。

## Mental Model

存包寄存：把每次都要带的“大件前缀”寄存在供应商那里，下次来直接取号复用，不用重新过安检；区别只在于有的柜子自动存（自动/隐式），有的要你手动办卡（显式）。

## Key Claims

- OpenAI 记录为自动缓存（explicit）。
- Anthropic 支持自动缓存和显式缓存（explicit）。
- Gemini 支持显式和隐式缓存，显式需手动创建（explicit）。
- 阿里百炼支持隐式和显式缓存，显式体验接近 Anthropic（explicit）。
- 火山自动缓存只支持自家模型，显式缓存对 Response API 友好，Context API 需手动维护（explicit）。
- LiteLLM 作为代理层可用于 prompt caching 和自动注入缓存 checkpoint（explicit）。
- 可比较维度：自动/隐式/显式、是否手动创建、API 维护成本、模型限制、代理层支持（inferred，供对比矩阵使用）。

## Examples

- `提示词缓存.xmind` 是一份火山/阿里百炼/Anthropic/OpenAI/Gemini/LiteLLM 的供应商对照笔记。
- 显式缓存普遍参照 Anthropic 的 checkpoint / cache-control 风格。

## Common Confusions

- 提示词缓存（请求间复用前缀）与 [[concepts/KV Cache|KV Cache]]（单次生成内复用历史 key/value）是不同层面的缓存。
- 缓存命中通常要求前缀稳定且对齐；动态变化的前缀难以命中。
- 这些供应商行为是时效性 API 能力，本页未联网核验，使用前需查官方文档。
