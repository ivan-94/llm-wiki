# 提示语工程

## Definition

提示语工程是通过组织指令、上下文、示例、约束和输出格式，引导 LLM 稳定完成目标任务的实践。

## Why It Matters

同一个模型在不同提示语下会表现出明显差异。好的提示语能减少歧义、明确输入输出契约、提高指令遵循度，并为后续评估和迭代提供可控变量。

## Mental Model

提示语不是一句话，而是一份任务接口文档：说明目标、输入、约束、过程、输出格式、失败处理和验收标准。

## Key Claims

- 提示语可以系统学习：结构框架、通用技巧、案例库、故障排除和场景化测试都是学习入口。
- 成熟系统提示语会包含身份、时间、知识边界、能力范围、不可执行事项、输出习惯和安全规则。
- 垂直场景 prompt 需要先建模任务类型和输出规范，例如公文写作要区分通知、会议纪要、议案等文体。
- 案例学习应抽取 prompt 的结构模式，而不是只复制原文。
- Agent 和长上下文场景会把提示语工程自然推向上下文工程。
- 提示语优化要通过固定数据集和指标验证，不能只依赖少数人工观感样例。
- 提示工程是一门经验科学，关键工作是建立评估、运行测试、识别失败模式并迭代 prompt。
- 高级 prompting 技巧可以按推理增强、任务分解、知识注入、工具调用、结构化输出和反思重写来组织。
- Agent 场景下，prompt 需要和工具 schema、控制流、状态管理、错误摘要、人类协作入口一起设计。

## Examples

- 系统 prompt 明确模型无法打开链接时，应要求用户粘贴内容。
- 多模态 prompt 明确面对人脸图片时不识别具体个人。
- “汉语新解”案例用类代码结构表达角色、函数、few-shot 和输出 SVG 卡片。
- 公文提示语先定义题材、输入字段、输出结构、格式规范和测试维度。
- Co-star、CRISPE、Camel 等框架都在帮助表达角色、上下文、目标、风格、语气、受众和响应格式。
- Chain-of-Thought、Tree-of-Thought、PAL、Least-to-Most、Prompt Chaining、RAG、ReAct 分别从推理、分解、程序辅助、检索和工具交互增强模型表现。

## Common Confusions

- 提示语工程不是“神奇咒语”，而是任务接口、上下文和评估标准的工程化表达。
- 好 prompt 不等于长 prompt；冗余说明会变成上下文噪声。
- 通用技巧不能替代场景建模，尤其是法律、公文、医疗、财务等格式和责任边界强的任务。

## My Understanding

现在的理解：提示语工程是给模型设计“任务接口”和“运行约束”。它不是一次性写提示，而是围绕评估、失败样例、示例选择、结构化输出和工具/状态边界持续迭代。
