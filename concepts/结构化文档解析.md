# 结构化文档解析

## Definition

结构化文档解析是把 PDF、图片、Office 文档等复杂资料解析成带布局、阅读顺序、表格、公式、图像和语义结构的中间表示，供检索、RAG 或 LLM 消费。

## Why It Matters

知识库质量常常卡在 ingest 前处理。只做 OCR 会丢失表格、版面、阅读顺序和图文关系；这些结构缺失会让 RAG 和总结结果失真。好的文档解析是"让 LLM 看到结构"而不是"让 LLM 读乱码"。

## Mental Model

结构化文档解析不是"读文字"，而是"重建文档"：先识别页面上的对象（文本/表格/公式/图像），再恢复对象之间的顺序和结构，最后生成可检索、可引用的结构化表示。

## 主流程

```
原始文档 → 布局识别 → OCR/文本提取 → 表格识别 → 公式识别 → 阅读顺序恢复 → 文档重建
```

## Key Claims

- 文档解析流水线通常包括布局识别、OCR/文本转换、表格识别、公式识别、图像裁剪和文档恢复。
- Document AI 需要结合文本、2D 位置、图像特征和布局信息；多模态模型（LayoutLMv3/Donut）比单模态模型准确率更高。
- explicit（来自结构化文档提取.xmind）：RVL-CDIP 对比：BERT 约 89%，DiT 约 92%，LayoutLMv3/Donut 等多模态 Transformer 约 95%。
- explicit（来自结构化文档提取.xmind）：Table Transformer 在 PubTables-1M 上表检测 AP 0.966、表结构识别+功能分析 AP 0.912。
- 面向 RAG 的解析应保留检索所需的块、表格、标题、阅读顺序和来源定位。

## DOCX Open XML 三层解析

`.docx` 不是单纯文本文件，而是 Open XML 包，需要分三层解析（来自 docx.xmind）：

| 层 | 内容 | 关键文件 |
| --- | --- | --- |
| **包结构层** | 发现所有部件（XML 文件、媒体、样式） | `[Content_Types].xml`、`_rels/` |
| **关系层** | 追踪部件间引用（图片、样式、超链接） | `_rels/*.rels` |
| **语义层** | 理解段落（`<w:p>`）、运行（`<w:r>`）、表格、图片、批注、修订 | `word/document.xml` |

- explicit（来自 docx.xmind）：仅抽取 `document.xml` 文本会丢失版式、列表层级、图片、链接和变更痕迹。
- inferred：面向 LLM/RAG 的 Word 解析应建立"部件发现 → 关系解析 → 语义重建"路径，而不只做正文文本抽取。

## 图像处理候选工具

对于图像文档（图片 PDF、扫描件）中的视觉预处理，Pillow 和 OpenCV 是 Python 侧常用候选（来自图像处理.xmind，信息量有限，具体流程见 `结构化文档提取.xmind`）。

## Examples

- `结构化文档提取.xmind` 比较 LayoutLMv3、DiT、PaddleOCR PP-Structure、Table Transformer、RAGFlow DeepDoc 等路线。
- `资料预处理.xmind` 汇总 PDF/Word/Excel/图片/网页转换：PDF 用 Marker 转 Markdown，Word 用 mammoth.js 转 HTML。
- `搜索.xmind` 给出搜索发现、页面抓取（Playwright/Crawl4AI）和正文抽取（Trafilatura/Readability.js）工具链。
- `音视频.xmind` 将音视频资料通过 VAD + ASR（Whisper/FunASR/SenseVoice）转换为字幕/文本。

## Common Confusions

- OCR 不是结构化文档解析的全部；表格单元格结构、阅读顺序和图文关系都需要额外处理。
- PDF 渲染（PDF.js）能看到页面，不等于提取了可用于 RAG 的结构（PDF.js 不提供结构信息）。
- `.docx` 解析不能只读 `document.xml`；图片、样式、编号列表在其他部件中，需要关系解析才能完整重建。

## My Understanding

结构化文档解析决定了下游 LLM 看到的是"乱序文字"还是"可追溯、有结构的资料"。DOCX 的三层解析模型（包结构 → 关系 → 语义）是一个可以迁移到其他 Office 格式的框架；PDF 的挑战在于"视觉即结构"需要用布局识别重建。
