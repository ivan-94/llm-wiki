# 上下文工程

## Definition

上下文工程是在 LLM 推理全过程中，对进入有限上下文窗口的指令、记忆、工具、知识、状态和历史进行策划、维护、选择、压缩和隔离的实践。

## Why It Matters

智能体越长时间运行，越容易积累错误、冗余、冲突和过时信息。上下文工程的目标不是把窗口填满，而是让模型在每一步只看到完成当前任务真正需要的信息。上下文工程 ≠ 提示语工程：提示语是上下文的一部分，但上下文还包括记忆、工具、知识、状态和历史。

## Mental Model

把 LLM 看成 CPU，把上下文窗口看成 RAM。提示语是程序入口，但智能体真正运行时还会把工具、状态、检索结果、历史和中间产物装进 RAM。上下文工程就是管理这块工作记忆——决定什么进、什么不进、什么压缩、什么隔离。

## 四类策略

| 策略 | 描述 | 示例 |
| --- | --- | --- |
| 写入外部 | 草稿、状态、长期记忆放在窗口外，需要时检索 | Agent 把中间结果写入文件而不是消息历史 |
| 选择 | 从草稿、记忆、工具库、知识库中挑当前任务所需内容；工具过多可用 RAG 做工具选择 | Code Agent 用 grep/AST 选相关代码而不是全部塞入 |
| 压缩 | 摘要、递归摘要、裁剪减少 token，只保留完成任务必要的信息 | Claude Code Auto Compact、会话历史递归摘要 |
| 隔离 | 多智能体、环境/沙箱、结构化状态隔离不同任务 | 子 Agent 只看自己任务的上下文，不接触主 Agent 的全局历史 |

## Key Claims

- 长上下文不自动等于高质量上下文；窗口越大，成本、延迟和注意力退化风险越高。
- 上下文失效常见形态包括投毒、干扰、混淆、冲突和腐烂（详见 [[concepts/上下文失效模式|上下文失效模式]]）。
- 工具选择也是上下文工程的一部分；过多工具定义会降低模型选择和执行质量，可用 RAG 做工具选择。
- JIT 检索通过在上下文中保留文件路径、数据库查询、网页链接等轻量标识符，并在需要时动态加载内容，减少常驻上下文负担（详见 [[concepts/JIT 检索（Just-in-Time Retrieval）|JIT 检索]]）。
- 长周期任务需要压缩、结构化笔记和子代理/多代理等机制来维护工作记忆和任务边界。
- 指令复杂度本身也是上下文质量问题：过度硬编码会脆弱，过度模糊会不稳定；少量高质量示例优于穷举边缘情况。
- 会话压缩与分层存储是上下文工程的实际部署形态（详见 [[concepts/会话分层存储与压缩|会话分层存储与压缩]]）。
- RAG 是上下文工程的"选择"策略之一，它帮助决定哪些知识进入当前窗口。

## Examples

- Code Agent 不把整个代码库放进 prompt，而是通过文件搜索、grep、AST、知识图谱和重排选择相关上下文（JIT 检索）。
- Claude Code 在上下文接近上限时触发摘要（Auto Compact），把完整轨迹压缩为后续可用的工作记忆。
- Agent 把临时草稿写入文件或状态，需要时再检索，而不是长期塞在消息历史里（写入外部）。
- 多智能体系统通过子 Agent 隔离不同任务的上下文，防止跨任务投毒（隔离）。
- LangChain History-aware RAG：在多轮对话中先改写追问，再检索，把会话依赖从检索阶段剥离（选择 + 压缩）。

## Common Confusions

- 长上下文模型不是上下文工程的替代品；它只是扩大了可管理的窗口，失效模式仍然存在。
- RAG 不是上下文工程的全部；它主要解决知识选择，上下文工程还包括工具、记忆、状态、历史和隔离。
- 多智能体不一定总是更好；它能隔离上下文，但会增加 token 成本和协调复杂度。
- 上下文工程和提示语工程不是对立关系：提示语是上下文的一部分，但上下文工程的范围更宽，包括所有进入窗口的信息。

## My Understanding

上下文工程是把"怎么问模型"扩展成"让模型在每一步看见什么、保留什么指针、压缩什么历史、隔离什么子任务"。它的难点在于取舍、边界和运行时维护，而不是堆信息。核心判断是：每一条进入上下文的信息，都要为模型的当前任务作出贡献——如果不贡献，就应该写到外部等待被检索。
