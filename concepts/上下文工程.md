# 上下文工程

## Definition

上下文工程是在 LLM 推理全过程中，对进入有限上下文窗口的指令、记忆、工具、知识、状态和历史进行策划、维护、选择、压缩和隔离的实践。

## Why It Matters

智能体越长时间运行，越容易积累错误、冗余、冲突和过时信息。上下文工程的目标不是把窗口填满，而是让模型在每一步只看到完成当前任务真正需要的信息。

## Mental Model

把 LLM 看成 CPU，把上下文窗口看成 RAM。提示语是程序入口，但智能体真正运行时还会把工具、状态、检索结果、历史和中间产物装进 RAM。上下文工程就是管理这块工作记忆。

## Key Claims

- 长上下文不自动等于高质量上下文；窗口越大，成本、延迟和注意力退化风险越高。
- 上下文失效常见形态包括投毒、干扰、混淆、冲突和腐烂。
- 上下文管理可以按写入外部、选择进入窗口、压缩已有内容、隔离不同任务四类策略组织。
- 工具选择也是上下文工程的一部分；过多工具定义会降低模型选择和执行质量。
- 多智能体、沙箱和结构化状态都可以被看作上下文隔离手段。
- JIT 检索通过在上下文中保留文件路径、数据库查询、网页链接等轻量标识符，并在需要时动态加载内容，减少常驻上下文负担。
- 长周期任务需要压缩、结构化笔记和子代理/多代理等机制来维护工作记忆和任务边界。
- 指令复杂度本身也是上下文质量问题：过度硬编码会脆弱，过度模糊会不稳定。

## Examples

- Code Agent 不把整个代码库放进 prompt，而是通过文件搜索、grep、AST、知识图谱和重排选择相关上下文。
- Claude Code 在上下文接近上限时触发摘要，把完整轨迹压缩为后续可用的工作记忆。
- Agent 把临时草稿写入文件或状态，需要时再检索，而不是长期塞在消息历史里。
- Agent 在上下文里保留文件路径或网页链接，等任务需要时通过工具读取具体内容，这是即时检索范式。

## Common Confusions

- 长上下文模型不是上下文工程的替代品；它只是扩大了可管理的窗口。
- RAG 不是上下文工程的全部；它主要解决知识选择，上下文工程还包括工具、记忆、状态、历史和隔离。
- 多智能体不一定总是更好；它能隔离上下文，但会增加 token 成本和协调复杂度。

## My Understanding

现在的理解：上下文工程是把“怎么问模型”扩展成“让模型在每一步看见什么、保留什么指针、压缩什么历史、隔离什么子任务”。它的难点在于取舍、边界和运行时维护，而不是堆信息。

## Review Questions

- 为什么长上下文会带来新的失效模式？
- 上下文投毒、干扰、混淆、冲突、腐烂分别在说什么？
- 写入、选择、压缩、隔离四类策略分别解决什么问题？
- 工具过多为什么也是上下文问题？
- JIT 检索和传统“把资料放进上下文”有什么差别？

## Open Questions

- 不同模型在长上下文下的具体退化曲线需要更多实证来源。
- 多智能体隔离与单 Agent 压缩之间的成本/质量边界还需要案例对比。
