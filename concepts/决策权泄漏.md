# 决策权泄漏

## Definition

决策权泄漏是指本该由人类、产品、领域规则、安全边界或架构约束决定的事项，被 Agent 在实现过程中静默决定。

## Why It Matters

泄漏的决策会被写进代码，表面上像实现细节，实际上改变了产品语义、权限模型、错误处理、数据状态或系统边界。

## Mental Model

决策权泄漏是 Agent 把“未定义”误当成“可以自由补全”。

## Decision Resolution Hierarchy (L0–L4)

不同层级的决策需要不同程度的治理：

| 层级 | 决策类型 | 示例 | 治理强度 |
| --- | --- | --- | --- |
| L0 | 产品/用户权益 | 谁可以删除哪类数据？ | 强制上浮，不可 Agent 静默决定 |
| L1 | 语义/领域规则 | "取消"是撤回还是删除？状态机？ | 必须在 spec/CONTEXT.md 中明确 |
| L2 | 架构/安全边界 | 缓存失效时用推还是拉？跨权限隔离？ | 需工程宪法或架构决策记录 |
| L3 | 接口/协议 | 错误码格式？分页策略？ | 有约定可放权，无约定需确认 |
| L4 | 实现细节 | 变量名、局部算法、内存分配 | 可完全放权给 Agent |

## Outcome Alignment

Outcome 具体化对齐原则：避免 spec 只写 Outcome（"登录成功"），而没有指定关键路径和边界（哪些 case 叫成功？失败如何分类？）。模糊 Outcome 等于默认 Agent 静默补全所有边界情况。

## Key Claims

- 权限例外、状态含义、错误码、校验层级和批量入口规则都可能是高层决策。
- 暴露所有决策不可行，因为决策具有分形特征；需要管理决策分辨率。
- L1 语义层和 L2 边界层应强治理，普通 L4 实现细节可以放权。
