# 子代理委派模式

## Definition

子代理委派模式是指主 Agent 把一个子任务派给隔离上下文、受限工具集的子代理执行，子代理只把最终摘要带回父上下文；它既是一种节省主上下文的执行结构，也是「子代理驱动开发」这种把探索/原型/评估外包给后台子代理的开发方式。

## Why It Matters

主 Agent 的上下文是稀缺资源。把噪声大、探索性强的子任务（读一堆文件、跑评估、做原型）交给子代理，只回收结论，能让主循环保持高信息密度。这也是后台 Agent 和复合工程的执行基石。

## Mental Model

子代理像「带回执的实习生」：你给它一个明确任务、一套受限工具和一个独立工位，它去折腾，最后只交一页结论；你不需要看它中间翻了多少文件。

## Key Claims

- `Hermes.xmind`：`delegate_task` 创建隔离上下文、受限工具集和独立终端会话的子 Agent，只把最终摘要带回父上下文。
- `Nano Bot .xmind`：子代理由 SpawnTool 异步创建，结果通过 [[concepts/Agent MessageBus|MessageBus]] 回灌为下一条入站消息，进入同一主循环。
- `Agent 解构.xmind`：多 Agent / 子代理需要 worktree 隔离、任务图、交叉验证、深度限制和最小系统提示，否则幻觉和依赖会放大。
- `提示词评估方案设计.xmind`：父 Agent 可 fork 多个子代理并发评估多个 domain，统一模型、提取低分项、汇总改进建议——子代理委派也用于并发评估。
- 子代理驱动开发：后台子代理负责探查代码库、在 worktree 中做原型、摸清解决方案空间，再把发现交回主线（与后台 Agent 阶段呼应，见 [[concepts/复合工程（Compound Engineering）|复合工程]]）。

## Examples

- 主 Agent 让子代理「在隔离 worktree 里实现并自测某 feature」，只回收「通过/失败 + diff 摘要」。
- 评估场景：一个父 Agent 并发派出多个评估子代理，每个评一个 domain，最后汇总低分项。

## Common Confusions

- 子代理委派不等于多 Agent 团队：前者是父子层级、单向回收摘要；后者是平级协作、双向通信（见 [[concepts/多 Agent 协作协议|多 Agent 协作协议]]）。
- 「只回收摘要」是关键设计，不是省事——目的是保护父上下文不被子任务噪声污染。
- 没有深度限制和隔离，递归委派会失控并放大幻觉。

## My Understanding

当前理解：子代理委派的精髓是「隔离上下文 + 受限工具 + 只回收摘要」；它让主 Agent 把脏活外包出去而不被脏活的中间过程拖累，是后台 Agent 和并发评估的执行单元。
