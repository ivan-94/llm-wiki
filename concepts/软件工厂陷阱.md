# 软件工厂陷阱

## Definition

软件工厂陷阱是把 AI Agent 接入代码库、任务系统和测试流程，却误以为这样就形成了高效软件工厂；实际上代码只是表面，背后的需求来源、推导过程、产品边界、权衡历史和验证标准——如果 Agent 看不到，评审者就要重新考古所有上下文。

## Why It Matters

Agent 可以快速写 diff、搭服务、补测试，但"生成正确代码"仍依赖需求、架构、业务约束和可维护性的证明；如果这些上下文不随代码一起结构化，Agent 的速度反而制造更多隐性负债。

## Mental Model

软件工厂 = Agent × 上下文包 × 可验证性 × 审查循环 × 权限边界；缺少任何一项，速度会把复杂度放大而不是消化。

## Failure Patterns

- **上下文考古成本**：Agent 改了文件，但评审者不知道改动来自哪个需求、依赖哪些假设、哪些 source 变化会让结果过期。
- **可验证性缺失**：不只需要知道改了哪些文件，还需要知道是哪些检查证明改动正确（proof type 必须匹配 claim type）。
- **上下文过载悖论**：更多上下文不等于更好结果；staleness detection 依赖"能限制上下文依赖范围"。
- **冲突假设**：并发 Agent 可能基于不同假设修改同一系统，合并时产生语义冲突。
- **proof type 错配**：API 200/后端单测/截图只能证明有限类型的结论；用错误的证据证明错误的 claim，系统看起来严谨但实际上是幻觉。
- **生成/验证速度差**：AI 生成变便宜而维护仍昂贵，如果没有准入、观测和删除机制，代码库会自然膨胀。

## Ontology & Epistemology

source 用两个框架组织这个问题：
- **Ontology（本体）**：建模 requirement、source、task、artifact、permission、test、proof claim、decision 等对象及关系。
- **Epistemology（认识论）**：要求 proof type 匹配 claim type。

## Key Claims

- explicit：AI 编程工具擅长写 diff、搭服务、补测试，但"生成正确代码"仍依赖需求、架构、业务约束和可维护性的证明。
- explicit：软件工厂不是 agent 本身，而是围绕 agent 的 onboarding、目标、权限、责任边界、任务定义、review loop、escalation path 与 DoD。
- explicit：proof type 必须匹配 claim type；错误类型的证据会让系统看起来严谨但实际上证明了错误的东西。
- explicit：更多上下文不等于更好结果；如果 agent 的输出可能依赖一切，staleness detection 几乎不可行。
- inferred：软件工厂陷阱的本质是把"生成速度"误认为"交付能力"，忽略了上下文、证据、审查和权限是工厂的必要组成。
- inferred：DeSlop 治理是走出软件工厂陷阱的代码库健康层，重点是让生成物带证据、付复杂度税，并有退场条件。

## Examples

- 没有 implementation notes 的 PR diff：评审者看到 400 行改动，不知道哪些是设计决策、哪些是偏差、哪些是权衡。
- 只用 "API 200" 证明业务规则正确：proof type（HTTP 状态码）无法证明 claim type（业务规则语义正确性）。

## Common Confusions

- 软件工厂陷阱不是说 AI Agent 不好用，而是说接入 Agent 本身不等于解决了工程问题。
- 解决方案不是减少 Agent，而是补上上下文包、可验证性、权限边界和审查循环。
