# 查询分析

## Definition

查询分析（Query Analysis）是在检索前对用户问题进行结构化处理的过程，包括查询分解、查询扩展、查询路由、问题改写、假设文档生成（HyDE）和结构化过滤器构造，以提升检索召回质量。

## Why It Matters

用户原始问题往往模糊、语境依赖、隐含多个子查询或不适合直接向量化检索。查询分析把问题转换成更适合检索系统消费的形态，是提升 RAG 准确率的关键前置步骤。

## Mental Model

把用户查询想象成一封手写地址的信：直接投递（向量化检索）可能送错，查询分析是把模糊地址转换成标准化地址（结构化查询）或拆成多个目的地（子查询）。

## Key Claims

- explicit（来自 Langchain 案例学习.xmind）：请求分析可用于 query decomposition（拆成子查询）、query expansion（扩展同义词和相关概念）、HyDE（先生成假设答案再检索）、路由、step-back 问题和结构化过滤器构造。
- explicit（来自 RAG vs 长文本.xmind）：即使使用长上下文方案，查询分析仍然必要。
- explicit（来自 Langchain 案例学习.xmind）：History-aware RAG 需要先把依赖聊天上下文的追问改写成独立问题，再执行检索。这是一种特殊的查询分析。
- inferred：HyDE（Hypothetical Document Embedding）通过让模型先生成一个假设答案文档，再用该假设文档的 embedding 检索，弥合问题和答案表达之间的语义差距。

## Examples

- 用户问"这个比上次贵多少"→ 查询分析改写为"当前价格是多少、上次价格是多少"两个独立子查询。
- Step-back 问题：把"如何用 Python 连接 PostgreSQL"改成"Python 数据库连接的常见方式是什么"，召回更广泛的参考资料。
- HyDE：先让模型生成"假设这道题的答案是……"，再用该假设文档向量化后检索，而不是直接向量化用户问题。
- SQL 场景查询分析：把自然语言问题转换成带有 filter、sort 和 select 的结构化查询。

## Common Confusions

- 查询分析不是为了替代检索，而是改善检索输入的质量。
- HyDE 虽然要额外调用模型，但在问题和答案在语义空间差距较大时有显著效果。
- 查询路由（routing）是把不同问题发送到不同检索后端，和 query decomposition 是不同操作。

## My Understanding

查询分析是"让检索听懂用户意图"的工程。用户的原始问题很少直接适合向量检索；把问题标准化、分解或扩展后，召回结果会显著改善。

- 查询分析本身会增加延迟；在实时 RAG 场景如何权衡需要实测。
