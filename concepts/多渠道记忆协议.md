# 多渠道记忆协议

## Definition

多渠道记忆协议是把「记忆」从单一聊天历史/单一向量库，扩展为「统一检索协议 + 多渠道适配器」的设计：对话、闪记、相册、日程、录音等异构渠道各自实现统一的 `search`/`readMore`/`installTools` 接口，由 Agent 按需决定查哪些渠道、读多深、用哪些专用工具。

## Why It Matters

真实个人助理的记忆分散在多个数据源，结构和检索方式各不相同。强行合并成一个向量库会丢结构、丢时间、丢专用查询能力。多渠道协议让每个渠道保留自己的检索逻辑，又给 Agent 一个统一入口和延迟决策能力，是设备级长期记忆的可扩展骨架。

## Mental Model

记忆像一栋楼里多个部门：每个部门（渠道）有自己的档案室和检索方式，但都开同一个服务窗口（统一协议）。Agent 像访客，先粗问（search），需要细节再深入（readMore），遇到特殊需求再借专用工具（installTools）。

## Key Claims

- `闪极记忆.xmind`（Loomos Memory）：记忆扩展机制支持三类能力——`search(input, filter)` 接收原始查询和过滤、返回 AI 可读记录；`readMore(referenceId)` 按引用读更深内容；`installTools()` 让渠道注册专用工具。
- 召回 filter 至少含 `user_id`、`start_datetime`、`end_datetime`，限定用户和时间范围。
- Record 含 AI 可读 content、channel、多模态 url/mime、references、原始 entity、created_at、score、user_id 等；`references` 支持多层索引（先返回标题/摘要，再 `readMore` 深读）。
- 渠道分动态记忆（对话，压缩时或用户显式请求时写入）和静态记忆（闪记、相册、日程、录音）；日程需按时间范围检索的自定义工具，录音可只向量化标题/描述/总结。
- Agent Loop 对多渠道记忆的价值是延迟决策：先粗召回，再按 references 深读，再用渠道工具处理特殊查询，而不必预先知道如何优化每个查询。

## Examples

- `ai_glass.xmind` 的「记忆管家」：先用最近聊天快速回忆判断，再做查询分析（分析时间范围 → 暴力检索该时段 → 生成关键词 → 向量检索），是多渠道/混合检索的早期形态。
- 录音渠道：向量库只放标题和总结，Agent 命中后通过 `readMore` 拉取转录详情。

## Common Confusions

- 多渠道记忆协议 ≠ 一个大向量库；它强调「每渠道自治 + 统一接口」。
- `references` 的多层索引不是为了省空间，而是为了让 Agent 先看摘要再决定是否深读，控制上下文成本。
- 协议层（search/readMore/installTools）与具体渠道实现（闪记/日程/录音）要分开看。

## My Understanding

当前理解：多渠道记忆协议的核心是「统一三件套接口（search/readMore/installTools）+ 渠道自治 + 多层引用延迟深读」，让 Agent 跨异构数据源做记忆检索而不被某一种存储绑死。
