# 反捷径证据

## Definition

反捷径证据是专门用来证明 Agent 没有通过投机、特判、浅层 happy path 或测试过拟合来“看似完成”的证据。

## Why It Matters

Agent 很擅长让当前测试通过，但这不等于语义正确。反捷径证据逼迫验收覆盖错误实现也可能通过的路径。

## Mental Model

反捷径证据要问：什么错误实现也能通过当前检查？

## Key Claims

- 强证据应覆盖规则族、组合空间、破坏性验证、参照系对比和对抗性审查。
- 反捷径审查应进入完成定义，而不是事后补充。
- 浏览器、HAT、review、LLM evaluator 和维护者审查都可以提供反捷径线索。

## Anti-Shortcut Review Checklist

错误实现也能通过测试的常见模式，用于审查清单：

- [ ] **条件覆盖不全**：测试只覆盖 happy path，Agent 可以把 else/edge case 写成任意代码。
- [ ] **类型宽松**：传 `any` 或 `string` 而非具体业务类型，实现可随意返回任意格式。
- [ ] **空实现**：测试不验证副作用（如数据库、消息队列），Agent 可以空实现通过。
- [ ] **常量投机**：Agent 直接硬编码预期返回值而不实现逻辑。
- [ ] **语义复制**：Agent 测试了某个值出现，但不测试它代表的业务含义。
- [ ] **隔离不足**：被测系统依赖全局状态，Agent 可以污染状态通过测试。
- [ ] **顺序依赖**：测试顺序执行时通过，但每个测试单独执行会失败。
