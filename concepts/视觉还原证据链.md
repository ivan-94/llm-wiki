# 视觉还原证据链

## Definition

视觉还原证据链是把“实现是否还原设计稿”从主观观感判断，转换为 target、token、state、screenshot、visual diff、patch、verification evidence 等可复查证据的工作流。

## Why It Matters

前端 Agent 很容易在视觉任务里凭感觉修，修完后又缺少可复查依据。证据链能让每次改动对应具体差异、截图和验证结果，避免越修越偏，或在响应式/交互状态下引入回归。

## Mental Model

```
Target -> Discipline -> Capture -> Diff -> Patch -> Verify
```

先冻结目标和规则，再按断点和状态捕获实现，用 diff 定位差异，修复后重新截图验证，直到重大差异收敛并留下证据。

## Key Claims

- 视觉还原不能只靠“看起来像”；每轮修复都应对应截图、diff、日志或 manifest。
- 输入应包含 Figma/目标截图、参考模板、标注说明、token、states 和 responsive 断点。
- 质量闸门包括标注确认、计划确认、素材可用、变量模式明确和视觉冲突处理。
- Visual fidelity loop 的证据至少覆盖 screenshots captured、diffs within threshold、responsive verified、states verified、no critical regressions。
- 有明确设计目标时应走视觉还原证据链；无目标设计稿而是审美改进时，应转入 [[concepts/对抗式 UI 审美审查|对抗式 UI 审美审查]]。

## Examples

- Figma 模板还原：先写 `annotation.md` 和 `sections.json`，人工确认后再实现模板，最后输出 preview artifacts 和 visual diff report。
- 多断点网页还原：分别截图 mobile、tablet、desktop，并验证 hover、active、disabled、focus 等状态。
- 单个按钮差异：不凭肉眼说“差不多”，而是用放大 diff 标出宽度、间距、颜色或阴影差异，再修复。

## Common Confusions

- visual diff 不是为了追求像素洁癖，而是让偏差来源可定位。
- 截图证据不能替代 token discipline；如果不先约束颜色、字体、间距和 radius，diff 只会暴露大量无意义差异。
- 响应式验证不是可选项；桌面像素对齐不能证明移动端没有破坏。

## My Understanding

视觉还原证据链的核心是“每次修复都要有差异来源和验证证据”，这样 Agent 才不会把审美任务做成无边界猜测。
