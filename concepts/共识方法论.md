# 共识方法论

## Definition

共识方法论把软件工程的本质定义为"在不同角色之间建立共享语义、明确边界、冻结决策、规范行为、可验证落地"的分层共识系统；在 AI Coding 时代，共识对象从"人与人"升级为"人与人 + 人与 AI"。

## Why It Matters

传统工程实践（PRD、DDD、接口文档、Code Review、TDD）从共识视角看，都是消除歧义的工具。进入 AI Coding 后，这些共识工具需要升级为 AI 可理解、可调用、可验证的形式，否则人虽然"说清楚了"，Agent 仍会在未定义空间自动补全。

## Mental Model

传统六层：需求共识→领域建模→接口共识→架构共识→工程行为共识→可执行共识。  
AI 升级后：每层由"人与人"扩展为"人 + AI"，共识载体从文档/约定变为上下文工程/本体/Contract/设计对齐/治理门禁/AI 验证。

## Six-Layer Framework

| 层级 | 传统方法 | AI 时代升级 |
| --- | --- | --- |
| 需求共识 | PRD、User Story、Acceptance Criteria | Context Engineering（背景、目标、边界、非目标、术语、约束、场景） |
| 领域建模 | DDD、Ubiquitous Language、Bounded Context | AI Shared Ontology（本体、词典、命名、类型、关系、枚举，版本化） |
| 接口共识 | API Contract、Interface-first Design | Prompt/Tool/Agent Contract（输入格式、规则、输出 Schema、能力边界、调用规范） |
| 架构共识 | ADR、RFC、Architecture Diagram | AI System Design Alignment（系统地图、架构图、数据流、依赖关系、长期记忆） |
| 工程行为共识 | Code Review、CI/CD、Definition of Done | AI Output Review & Governance（质量门禁、安全、风格、复杂度、成本治理、人在回路） |
| 可执行共识 | Test、TDD、Formal Specification | AI Verification（Eval、Benchmark、Golden Test、Regression Suite、日志、追踪） |

## Key Claims

- explicit（图）：软件工程的本质被定义为共识工程，目标是消除歧义、提高效率和质量、支持持续演化。
- explicit（图）：AI Coding 变革的方向是从"人与人达成共识"升级为"人与人 + 人与 AI 达成共识"。
- explicit（图）：隐式信息（上下文停在聊天/术语约定/规则靠经验/边界靠感觉）需要升级为显式信息（上下文完整/术语已定义/规则清晰/边界严格）。
- inferred：共识方法论为 AI Coding 提供了方法论框架：哪些共识必须显式化，才能让 Agent 不在该层自动补全。

## Examples

- Context Engineering 作为"需求共识"的 AI 升级形态。
- `Prompt/Tool/Agent Contract` 作为接口共识的 AI 形态，覆盖输入/输出/能力边界/副作用。
- `Golden Test + Regression Suite` 作为可执行共识的 AI 验证工具。

## Common Confusions

- 共识升级不是把文档写得更长，而是把关键语义、规则、边界变成 AI 可消费的结构化输入。
- 并非所有六层都要同等升级；快速原型可以只升级 L1 需求共识；严肃项目至少要升级到 L4。

## My Understanding

当前理解：把"软件工程实践"重命名为"分层共识系统"后，AI Coding 的工具配置就有了明确骨架：每层共识需要什么载体，哪些载体能被 AI 消费、调用、验证。
